# Figma 파일 구조 & 시안(Concept) 워크플로우 가이드

디자인팀(design-pl 이하 전원)이 Figma 파일에 페이지를 만들 때 따르는 공통 규칙이다. `report-style.css`/`report-format-guide.md`가 doc-writer의 공유 템플릿이듯, 이 문서는 Figma에 접근하는 모든 에이전트(design-scanner, brand-designer, graphic-designer, design-systems, ux-designer, ui-designer, interaction-designer, motion-designer, content-designer, design-qa)의 공유 템플릿이다.

## 0. Figma 조회 결과를 그대로 믿지 않는다 (알려진 도구 한계 — 2026-07-12 확인)

`get_metadata`를 nodeId 없이 호출하면 "최상위 페이지 목록"을 반환하는데, **이 목록이 실제로 존재하는 페이지를 전부 보여주지 않는 경우가 실제로 확인됐다** (페이지가 여러 개 있는데 1개만 반환된 사례). 그래서 **이 목록 하나만 보고 "이 페이지가 없다"고 단정하면 안 된다.**

- 특정 페이지(예: "레퍼런스")를 찾는데 최상위 목록에 안 보여도, **이전 대화·메모리·사용자 지시에 그 페이지의 이름이나 nodeId가 이미 언급되어 있으면**, 목록을 믿지 말고 그 nodeId로 직접 `get_metadata`/`get_screenshot`을 호출해 실제로 있는지 재확인한다.
- 그렇게 확인해도 못 찾겠으면, 존재하지 않는다고 단정하고 **조용히 다른 자료로 대체해서 진행하지 않는다.** design-pl에게 "OO를 못 찾았다"고 명확히 보고하고, design-pl이 사용자에게 확인을 요청하게 한다.
- 특히 **사용자가 직접 준비해둔 자료(레퍼런스, 무드보드 등)가 관련된 작업**에서는 이 원칙을 특히 엄격히 지킨다 — 없다고 판단하고 다른 근거(예: 기획 문서)로 조용히 대체하면, 사용자의 실제 의도와 다른 산출물이 나온다.

**`.claude/agent-memory/`와 `docs/design/`의 역할은 다르다 — 절대 섞지 않는다**:
- `.claude/agent-memory/{에이전트}.md`는 **휘발성 세션 로그**다. "최근에 무슨 작업을 했는지"만 남기고, 항목이 5개를 넘으면 오래된 것부터 지운다(git 히스토리가 원본이라 지워도 유실 없음). **여기에 프로젝트의 지속적인 진실 상태(브랜드 결정, 토큰/컴포넌트 인벤토리, 확정 스펙)를 저장하지 않는다** — 이건 다른 세션·다른 에이전트가 재사용할 근거가 아니라 이 에이전트 개인의 최근 맥락 메모다.
- `docs/design/*.md`는 **지속되는 진실 상태(source of truth)**다. 프로젝트가 끝날 때까지, 세션이 몇 번을 넘어가든 항상 최신 상태를 반영해야 하는 문서라서 캡 규칙이 없다(상태 문서는 덮어쓰기, 이력 문서는 append). 재사용 가능한 하네스이므로, 다음 세션의 어떤 에이전트든 memory를 참고하지 않고 이 파일 경로 하나만으로 전체 맥락을 복원할 수 있어야 한다.

**스타일 소스 오브 트루스** (전부 `docs/design/` 아래): 브랜드 컬러·톤은 `docs/design/brand-guide.md`(brand-designer가 덮어써서 관리), 아이콘/오브제 원화는 `docs/design/graphic-assets.md`(graphic-designer가 덮어써서 관리), 토큰/컴포넌트 목록은 `docs/design/design-system.md`(design-systems가 덮어써서 관리)가 단일 출처다. 모든 에이전트는 자기 판단으로 색상·값·아이콘을 새로 정하지 않고 이 파일들을 먼저 읽고 시작한다 (이미 각 에이전트 정의에 명시되어 있음).

**확정 스펙 문서(`docs/design/confirmed/`)는 위 세 문서보다도 더 구체적인 근거다** (2-3번 참고) — 3개 문서가 헤드라인 요약(컬러 hex, 폰트명 정도)이라면, 확정 스펙 문서는 확정된 프레임을 직접 다시 본(get_screenshot/get_design_context) 사람이 spacing·radius·그림자·레이아웃 관례까지 적어둔 더 풍부한 기록이다. 실제로 "브랜드 톤은 반영됐는데 확정한 시안의 레이아웃/디테일은 안 반영됐다"는 문제가 헤드라인 요약만으로 작업해서 생긴 적이 있다. 두 소스가 다르면 확정 스펙 문서를 따른다.

## 1. 페이지 순서 (Figma 파일 안의 "디렉토리") 및 분야별 네이밍 규칙

```
1. Cover                        — 표지 (필요시)
2. 레퍼런스                      — 사용자가 직접 관리하는 무드보드. 팀은 여기 쓰지 않는다.
--- BRAND CONCEPTS ---
3. "Brand Guide"                 — brand-designer 전용 (로고/컬러/타이포/톤 + 일러스트레이션·오브제 스타일 방향)
4. "{작업명} Concepts"            — 시안 3개가 모이는 공용 페이지 (아래 2번 참고). Brand Guide와 같은 구역에
                                     둔다 — 브랜드 방향을 정하는 작업과 그 시안이 논리적으로 같은 초기 단계이기 때문이다
                                     (2026-07-14, 사용자가 두 구역을 직접 합쳐서 확정).
--- GRAPHIC ASSETS ---
5. "Graphic Assets"               — graphic-designer 전용. 아이콘/오브제/애니메이션용 소스 그래픽의 원화.
                                     design-systems·motion-designer·ui-designer·content-designer가 여기서 완성된
                                     그래픽을 가져가 각자 정식 자산(컴포넌트/애니메이션/화면/마케팅 비주얼)으로 만든다.
--- FOUNDATIONS ---
6. "Colors" / "Typography" / "Spacing" / "Elevation" / "Icons" — design-systems 전용, 토큰·에셋 카테고리별 1페이지
                                     ("Icons"는 graphic-designer 원화를 컴포넌트 variant로 정식 등록한 결과)
--- COMPONENT SPECS (문서화, 2-5번 참고) ---
7. "Component Specs"              — design-systems 전용, 사람이 클릭 없이 훑어볼 수 있는 컴포넌트별 스펙 시트(제목+설명+전체 variant 그리드)를 모아두는 **단일 공용 페이지**(Concepts 페이지처럼 여러 항목을 한 페이지에 나란히 배치 — 컴포넌트마다 새 페이지를 쪼개지 않는다). State/Focus 등 variant가 있는 컴포넌트를 등록하거나 갱신할 때마다 이 페이지에 해당 스펙 시트를 추가/갱신한다(2-5번 규칙).
--- COMPONENTS ---
8. "{컴포넌트명}" (예: "Button")   — design-systems 전용, 컴포넌트 하나당 페이지 하나(실제 컴포넌트 정의·인스턴스가 사는 곳). "Component Specs" 페이지는 이 개별 페이지를 대체하지 않는다 — 개별 페이지는 컴포넌트 원본이 사는 곳이고, Component Specs는 그걸 훑어보기 좋게 모아둔 별도 문서화 뷰다.
--- SCREENS ---
9. "{카테고리명}" (예: "Auth", "Contacts") — **화면 하나당 페이지 하나가 아니다.** 관련된 화면들을
                                     기능 카테고리로 묶어 페이지 하나에 나란히 배치한다(예: 로그인/
                                     회원가입/비밀번호 재설정은 전부 "Auth" 페이지 안에, 연락처
                                     목록/추가·수정/빈 상태는 전부 "Contacts" 페이지 안에). 화면마다
                                     페이지를 새로 만들면 화면이 늘어날수록 페이지를 계속 넘나들어야
                                     해서 비교·검토가 힘들어진다 — 실제로 이 문제가 지적됐다.
                                     ui-designer가 레이아웃을 만들고, interaction-designer가
                                     **같은 페이지 안에서** 프로토타입 연결(전환 애니메이션)을 추가한다.
                                     화면과 그 화면의 인터랙션은 페이지를 분리하지 않는다.
                                     **파일럿 단계는 예외적으로 그 자체가 하나의 카테고리다** — 파일럿
                                     화면 전부를 "파일럿" 페이지 하나에 나란히 배치한다(2번의 Concepts
                                     페이지와 같은 방식). 카테고리가 확장되면(3-B번 전체 확장 단계)
                                     그때 정식 카테고리 페이지로 옮긴다.
--- MOTION ASSETS (필요할 때만) ---
10. "Motion Assets"                — motion-designer 전용. graphic-designer의 레이어 분리된 소스를 받아
                                     실제 애니메이션(타이밍/easing/Lottie)을 완성한다. 프레임 이름은
                                     "{에셋명} — {유형}" (예: "Spinner — Loading", "Logo — Animated Icon")
--- MARKETING (필요할 때만) ---
11. "Marketing — {캠페인/용도명}"  — content-designer 전용 (배너/랜딩/SNS 카드). 커스텀 그래픽이 필요하면
                                     graphic-designer에게 소스를 요청해 받아 쓴다.
```

**Figma 파일 반영 확인(2026-07-14, design-systems)**: 사용자 요청으로 실제 Figma 파일(`zgGlMBwFglaDlaeyP4CkgR`)의 페이지 목록에도 위 구역 구분 스타일을 그대로 적용했다 — 구역 경계마다 같은 이름(`--- BRAND ---` 등)의 빈 구분용 페이지를 만들고, 전체 페이지 순서를 위 문서 순서에 맞게 재배열했다(아직 실제로 쓰이는 구역이 없는 `--- GRAPHIC ASSETS ---`/`--- MOTION ASSETS ---`/`--- MARKETING ---`은 스킵).

**구역 병합 안내(2026-07-14, 사용자 직접 작업)**: 사용자가 Figma에서 `--- BRAND ---` 구역과 `--- CONCEPTS ---` 구역 안의 페이지를 직접 `--- BRAND CONCEPTS ---` 하나로 합쳤다. 이 문서(1번 항목)는 그 결정을 그대로 반영해 두 구역을 병합했다 — 앞으로 이 팀이 만드는 어떤 프로젝트에서도 Brand Guide와 Concepts 페이지는 기본적으로 같은 구역에 인접 배치한다. 최신 페이지 순서 전체(신규 구분 페이지 ID 포함)는 `docs/design/design-system.md` 8절 "페이지 순서" 표를 참고한다 — 단, 이 병합은 사용자가 Figma에서 직접 한 작업이라 그 표는 아직 이 병합을 반영하지 못했을 수 있으니, 다음에 design-systems가 이 구역을 다루게 되면 실제 Figma 상태를 다시 확인하고 표를 갱신한다.

각 에이전트는 자기 담당 구역에만 페이지를 만든다. 다른 에이전트 구역에 페이지를 만들지 않는다. `interaction-designer`만 예외로, 자기 전용 페이지를 따로 만들지 않고 `ui-designer`가 만든 화면 페이지 위에서 작업한다 (인터랙션은 화면과 분리될 수 없는 속성이기 때문).

## 2. 시안(Concept) 워크플로우 — 처음 방향을 정할 때만

**새로운 컨셉/방향을 처음 정하는 작업**(예: 브랜드 방향, 컬러 팔레트, 특정 화면의 레이아웃, 아이콘/오브제 스타일을 맨 처음 잡을 때)은 최종본 1개를 바로 만들지 않는다. 대신:

1. 담당 에이전트가 서로 다른 방향의 **시안 3개**를 만들어 `"{작업명} Concepts"` 페이지에 나란히 배치한다. **"서로 다른 방향"은 컬러/퍼스낼리티만 다른 게 아니라, 실제로 구분되는 디자인 트렌드와 레이아웃 구조까지 달라야 한다** (2-1번의 트렌드/레이아웃 다양성 기준 참고).
2. 각 시안 옆에 짧은 설명(이 시안의 컨셉·의도, 1~2문장)을 붙인다.
3. design-pl이 3개 시안을 갖춘 뒤 메인 세션에 요약을 반환하고 멈춘다. 메인 세션이 사용자에게 보여주고 확정을 요청하며, 실제 확정 결과만 정해진 신뢰 형식으로 design-pl에게 다시 전달한다. **이 형식의 사용자 확정 없이 다음 단계로 진행하지 않는다.**
4. 확정되면, 확정된 방향의 성격에 따라 다음 표준 파이프라인을 순서대로 진행한다(중간 단계를 건너뛰지 않는다):
   - **브랜드/톤앤매너 컨셉이 확정된 경우**: 1차 확정 직후 바로 (a)로 넘어가지 않는다 — 먼저 2-1번의 **2차 라운드**(레이아웃/구성/트렌드 정밀화, 실제 화면 요건 반영)를 거쳐 그 결과까지 별도로 확정받는다. 2차까지 확정되면 다음 순서를 지킨다 — **"시스템을 먼저 만들고 조립"이 아니라 "확정된 실제 디자인을 먼저 완성하고 거기서 시스템을 추출"하는 순서다** (근거: 3-B번):
     - (a) brand-designer가 확정된 1차/2차 프레임을 직접 다시 보고 "Brand Guide" 페이지로 정식화(2-3번 기준의 디테일까지 포함, 일러스트 스타일 방향 포함).
     - (b) design-prompter가 2-3번에 따라 확정 스펙 문서(`docs/design/confirmed/`)를 작성.
     - (c) ui-designer가 확정된 2차 시안(예: B-2)을 **그대로 이어받아 발전**시켜(재해석·재설계가 아니라 elevate) SCREENS 구역에 파일럿 화면을 프로덕션 완성도로 제작(3-B번). 이 시점엔 design-systems 토큰이 아직 없으므로 확정 스펙 문서/Brand Guide의 원본 값을 직접 쓴다. 아이콘이 필요하면 graphic-designer에게 요청해 그 자리에서 받아 쓴다.
     - (d) 사용자가 파일럿을 승인하면(3-B번 피드백 루프 통과), **design-systems가 그 승인된 파일럿에서** 토큰·컴포넌트·아이콘 세트를 추출(extract)해 정식 등록한다 — 파일럿과 무관하게 독립적으로 새로 설계하지 않는다.
     - (e) design-qa 감사.
   - **팔레트/토큰만 확정된 경우**: design-systems가 정식 토큰/컴포넌트 생성 → ui-designer 화면 적용 → design-qa.
   - **화면 레이아웃만 확정된 경우**: ui-designer가 SCREENS 구역에 정식 레이아웃 완성 → design-qa.
   - **아이콘/오브제 스타일만 확정된 경우**: graphic-designer가 "Graphic Assets"에 정식 세트 완성 → design-systems가 기능 아이콘을 "Icons"에 컴포넌트로 등록 → (필요시) ui-designer 화면 적용 → design-qa.
5. 채택되지 않은 시안은 삭제하지 않는다 — `"{작업명} Concepts"` 페이지에 이름 앞에 `❌ 미채택 —`을 붙여 그대로 남겨둔다 (의사결정 히스토리 보존, 나중에 "왜 이 방향이 아니었는지" 참고 가능). **시안 세트가 여러 에이전트의 프레임으로 구성된 경우**(예: brand-designer의 톤 프레임 + ui-designer의 적용 화면 프레임이 한 세트), 4번의 정식화 작업을 맡은 에이전트가 그 세트에 속한 모든 프레임(다른 에이전트가 만든 것 포함)에 `❌ 미채택 —` 라벨을 붙인다 — Concepts 페이지는 공용이므로 라벨링(이름 변경)은 내용 수정이 아니라 관리 작업으로 간주한다. 시각적 내용 자체는 원작성자 동의 없이 바꾸지 않는다.
   - **이 규칙에는 예외가 없다.** 이전 시안이 요건을 놓쳤거나 버그가 있었던 "재작업"이라도 마찬가지다 — "이건 진짜 대안이 아니라 실수였다"는 이유로 조용히 삭제하거나 덮어쓰지 않는다. 컨셉(1차 브랜드/방향), 1차 시안, 2차 시안(2-1번) 모두 같은 방식으로 원본을 남기고 `❌ 미채택 —` 라벨만 붙인다 — 이유가 "다른 대안이라서"든 "결함이 있어서"든 취급은 동일하다.

**시안이 필요 없는 경우**: 이미 확정된 방향을 후속 작업에 그대로 적용할 때(예: 로그인 화면 레이아웃이 확정된 뒤 관리 화면을 만들 때 같은 컴포넌트를 재사용하는 것)는 매번 시안 3개를 새로 만들지 않는다. 시안 단계는 **새 방향을 정할 때 한 번**만 거친다.

### 2-1. 브랜드/톤앤매너 컨셉 시안은 기본적으로 "적용 화면" 세트로, 트렌드·레이아웃이 서로 다르게 만든다

브랜드 방향(톤앤매너, 컬러, 로고 등)을 처음 정하는 라운드는 컬러 스와치·타이포 샘플만으로 끝내지 않는다. 사용자가 톤만 보고는 실제로 어떤 느낌인지 판단하기 어렵기 때문에, **기본값으로** 다음 두 가지를 세트로 함께 만든다 (사용자가 매번 "화면도 보여달라"고 다시 요청할 필요 없이 design-pl이 계획 단계에서 먼저 포함한다):

1. **brand-designer**가 만드는 톤 프레임(컬러/타이포/워드마크 톤 예시) — 기존과 동일. 이때 3개 시안은 각각 **서로 다른 실제 디자인 트렌드/무브먼트**에 명시적으로 대응시킨다(예: 미니멀·스위스 그리드, 뉴브루탈리즘, 글래스모피즘/뉴모피즘, 벤토 그리드, Y2K·레트로퓨처리즘, 클레이모피즘, 스큐어모피즘 등 실제 통용되는 스타일 중 제품 성격에 맞는 것). 색만 다르고 같은 트렌드 안에서 변주하는 3개를 만들지 않는다. 각 시안 설명에 어떤 트렌드에 대응하는지 명시한다.
2. **ui-designer**가 만드는 대표 화면 1장(예: 메인 화면) — 그 컨셉의 원본 색상/폰트 값을 화면에 직접 적용한 가벼운 목업. 이 단계에서는 design-systems를 투입하지 않는다(아직 확정 전이라 토큰화 대상이 없음). 새 페이지를 만들지 않고, brand-designer가 만든 해당 컨셉의 톤 프레임 옆에 배치한다(같은 `"{작업명} Concepts"` 페이지 — 공용이므로 가능). 이때 3개 화면은 네비게이션 위치(상단바/좌측 사이드바 등), 리스트 표현 방식(카드형/테이블형/타임라인형/그리드형), 정보 밀도가 실질적으로 달라야 한다 — 같은 골격에 색만 다르게 칠하지 않는다. brand-designer가 지정한 트렌드에 맞는 레이아웃 관례를 따른다(예: 뉴브루탈리즘이면 굵은 테두리·비대칭 그리드, 미니멀이면 넉넉한 여백·단일 컬럼 등).

이 두 단계는 design-pl이 순차 호출한다(brand-designer 먼저 → ui-designer). 3개 컨셉 모두에 이 세트가 갖춰진 뒤에 design-pl은 메인 세션에 반환하고 멈추며, 메인 세션이 사용자에게 보여주고 확정을 요청한다. 이것이 **1차 확정**이다.

### 2-1-B. 1차 확정 후 — 2차 라운드(레이아웃/구성/트렌드 정밀화)

1차의 "대표 화면 1장"(2-1번 2항)은 브랜드 톤 자체를 비교·판단하는 게 목적이라 레이아웃이 단순할 수 있다. 1차에서 브랜드 방향이 하나로 확정됐다고 곧바로 정식 파이프라인(Brand Guide 정식화 등)으로 넘어가지 않는다 — 확정된 그 방향 하나로, 실제 대상 화면의 **레이아웃/화면구성/트렌드 표현이 서로 다른 시안 3개**(2차 시안)를 다시 만들어 정밀 비교하는 라운드를 한 번 더 거친다.

- 담당: design-prompter(브리프) → ui-designer 단독 호출. design-systems는 아직 투입하지 않는다(확정된 브랜드의 원본 값을 그대로 사용 — 이미 정식 토큰이 있다면 토큰을 우선한다).
- **1차와의 차이**: 1차는 "이 톤이 맞는 방향인가"를 보기 위한 것이라 화면 요소가 간단해도 된다. 2차는 "이 방향으로 실제 화면을 어떻게 짤 것인가"를 보기 위한 것이므로, `docs/planning`에 정의된 실제 요소(검색/필터, 목록 필드, 액션, 상태 등)를 반드시 반영한다 — placeholder로 대체하지 않는다.
- 3개 시안은 네비게이션 위치, 리스트/콘텐츠 표현 방식, 정보 밀도, 트렌드 표현이 서로 실질적으로 달라야 한다(2-1번 1항과 동일 기준을, 이번엔 브랜드를 하나로 고정한 채 적용).
- **네이밍 규칙**: 2차 시안은 A/B/C를 새로 쓰지 않는다 — 1차에서 확정된 시안의 라벨을 그대로 이어받아 접미사 `-1`/`-2`/`-3`만 붙인다. 예: 1차에서 "Concept B"가 확정됐으면 2차 시안은 `B-1`/`B-2`/`B-3`로 이름 붙인다. 이렇게 하면 어떤 1차 결과에서 파생된 2차인지, "그 B가 이 B인지" 매번 헷갈리지 않는다.
- 배치: 새 페이지를 만들지 않고, 같은 `"{작업명} Concepts"` 페이지 안에 1차 시안 근처(예: 아래쪽)에 2차 시안 3개를 추가한다.
- 2차 시안 3개가 갖춰지면 design-pl은 다시 메인 세션에 반환하고 멈춘다. **2차 확정은 1차 확정과 별개의 게이트다** — 2차까지 사용자가 확정해야 정식 파이프라인으로 진행한다.
- Type B(팔레트만)·Type C(화면 레이아웃만) 라운드는 이 2차 단계를 거치지 않는다 — 그 라운드의 1차 시안 자체가 이미 실제 요건을 반영한 레이아웃/토큰 시안이기 때문이다(2번 1항).

### 2-2. 아이콘·일러스트레이션(오브제) 담당 — 누가 만드는지

기능 아이콘(검색/추가/수정/삭제/카테고리 등)과 장식용 오브제(마스코트, 빈 상태 그래픽, 온보딩 일러스트 등)는 다음과 같이 나눠 담당한다. Atomic Design 관점에서 아이콘도 컴포넌트와 같은 atom/molecule 단위 에셋이지만, **그리는 것(크래프트)과 시스템에 등록하는 것(토큰/컴포넌트화)은 서로 다른 기술이라 담당을 분리한다.**

- **brand-designer**: 일러스트레이션/오브제의 스타일 **방향**만 Brand Guide에 정의한다(라인/플랫/3D/아이소메트릭 등, 브랜드 톤과의 일관성 근거 포함). 실제 개별 에셋은 만들지 않는다.
- **graphic-designer**: brand-designer가 정한 방향을 따라 **아이콘/오브제 원화를 실제로 그린다** — 이 팀에서 그래픽 크래프트를 전담하는 유일한 에이전트다. "Graphic Assets" 페이지에 아이콘 세트와 장식 오브제를 완성하고, 애니메이션이 필요한 것은 motion-designer가 다루기 쉽도록 레이어를 분리해둔다. 토큰화·화면 조립·애니메이션 자체는 하지 않는다.
- **design-systems**: graphic-designer가 완성한 기능 아이콘을 가져와 FOUNDATIONS 구역의 `"Icons"` 페이지에 컴포넌트와 동일한 방식(변형·스크린샷 검증)으로 **정식 등록**한다. 아이콘을 직접 새로 그리지 않는다 — graphic-designer의 결과물을 시스템화하는 역할이다.
- **ui-designer**: design-systems가 정식 등록한 Icons 세트와, graphic-designer가 그리고 brand-designer가 방향을 정한 오브제를 그대로 가져와 화면에 조립한다(임의로 새 아이콘/일러스트를 만들지 않는다).

**토글/짝 상태 아이콘은 세트로 완성해야 등록 완료다**: 확정 프레임은 한 시점의 스냅샷이라, "비밀번호 표시/숨김 토글"처럼 기능적으로 두 가지 이상의 모습이 필요한 아이콘도 그 순간의 모습 하나만 그려져 있는 경우가 많다(예: `Pixel/Eye` — "표시/숨김 토글"이라고 설명은 붙어있는데 실제로는 눈 모양 하나만 등록됐던 사례, 2026-07-14 발견). design-systems가 확정 프레임에서 아이콘을 clone/추출할 때, 그 아이콘이 토글·체크/언체크·펼침/접힘 같은 **다중 상태 기능**을 나타낸다면, 한쪽 모습만 있고 나머지가 없는 채로 등록을 완료로 보고하지 않는다 — 없는 쪽은 **graphic-designer에게 요청해서 같은 스타일 트랙(strokeWeight, 사각 블록 조합 등 기존 아이콘 세트의 조형 관례)으로 새로 그려서** 세트를 완성한 뒤에만 정식 등록한다. 이건 2-4번 "빠진 아이콘"(레이아웃에 필요한데 아예 없는 아이콘) 규칙과 같은 원리를 "짝이 되는 상태"에도 적용한 것이다.
- **motion-designer**: graphic-designer가 레이어 분리해둔 소스 그래픽을 받아 실제 애니메이션(타이밍/easing/Lottie)을 완성한다. 그래픽 자체를 새로 그리지 않는다.
- **content-designer**: 마케팅용(배너/랜딩/SNS) 비주얼에 한정된다 — 제품 화면(로그인, 목록 화면 등)에 들어가는 아이콘·오브제는 이 에이전트의 범위가 아니다. 마케팅 비주얼에 커스텀 그래픽이 필요하면 graphic-designer에게 소스 제작을 요청해 받아 쓴다(직접 그리지 않는다).

**등록된 아이콘은 실제로 화면에서 쓰여야 한다 — 만들어놓고 텍스트로 때우지 않는다**: "Icons" 페이지에 정식 등록된 기능 아이콘이 있는 액션(수정/삭제/검색/카테고리/로그아웃 등)을, 화면에서 "수정"/"삭제" 같은 텍스트 버튼으로만 처리하고 아이콘을 아예 안 쓰는 건 에셋 낭비이자 완성도 저하다 — 실제로 이런 누락이 있었다. ui-designer는 화면을 만들기 전 `docs/design/design-system.md`의 Icons 목록을 확인하고, 그 목록에 해당하는 액션이 화면에 등장하면 텍스트가 아니라 그 아이콘을 사용한다(design-qa도 이 항목을 감사한다).

**단, "쓴다"가 "억지로 다 채운다"는 뜻은 아니다 — 최종 판단은 시각적 결과다**: 등록된 아이콘이라도 자리가 안 어울리거나(예: 슬롯 자체가 없는 자리에 우겨넣기), 실제로 배치했을 때 비율이 깨지거나 겹치는 등 결과가 텍스트보다 나빠지면 아이콘을 쓰지 않는 게 맞다 — 판단 근거만 남기면 된다(실제 사례: 사이드바 카테고리 nav에 아이콘 슬롯이 없어 보류, 행 액션 버튼은 아이콘 전용으로 바꿨다가 사용자가 이전 텍스트+색 버전이 더 낫다고 판단해 되돌림). "아이콘을 썼는가"가 체크리스트 통과 기준이 아니라 "화면이 실제로 더 나아졌는가"가 기준이다.

### 2-3. 확정될 때마다 "확정 스펙 문서"를 남긴다 — 컬러 hex 몇 개로 요약하지 않는다

1차든 2차든 사용자가 뭔가를 확정하면(신뢰 형식으로), 그 순간 이후 작업하는 모든 워커가 "확정된 게 정확히 뭔지" 같은 근거로 일해야 한다. 컬러 hex와 폰트명 정도의 헤드라인 요약만으로는 부족하다 — 간격 리듬, 모서리 반경, 그림자 유무, 아이콘/장식 모티프, 레이아웃 구조의 세부 관례처럼 "그 시안을 실제로 좋아 보이게 만든" 디테일이 요약 과정에서 빠지면, 뒤 단계 워커가 자기 해석으로 다시 만들어버리고 컨셉이 무너진다.

- **누가 관찰하는가**: 확정된 프레임을 실제로 만든/책임진 에이전트(1차 브랜드 톤이면 brand-designer, 2차 레이아웃이면 ui-designer)가 `get_screenshot`/`get_design_context`로 **그 확정 프레임을 다시 직접 본다** — 이미 갖고 있는 요약 메모리를 재활용하지 않는다. 색상/타이포뿐 아니라 spacing 리듬, radius, 그림자, 아이콘·장식 모티프, 레이아웃 세부 관례(정렬, 카드/행 간격, 강조 방식 등)까지 관찰한 그대로 기록한다.
- **자로 잴 수 있는 것과 컨셉의 성격을 표현하는 것은 서로 다른 관찰 대상이다 — 둘 다 기록해야 한다.** spacing/radius/border 두께 같은 구조적 수치만 기록하고 끝내면, "네오팝 배지형 — 다정하지만 경계는 또렷하게" 같은 컨셉의 실제 성격이 "선을 두껍게 긋는다"는 밋밋한 규칙 하나로 쪼그라드는 사고가 실제로 있었다. 브랜드 톤 프레임(1차 원본, 2차 레이아웃이 아니라)을 별도로 다시 보고, 그 컨셉을 컨셉답게 만드는 표현 장치(배지·스티커 같은 형태 은유, 워드마크의 정확한 레터링/처리 방식, 대비를 통해 성격의 이중성을 드러내는 방법 등)를 구조적 수치와 나란히 기록한다. 이 장치들은 이후 로고 자리·장식 요소에 그대로 적용돼야 하며, 빠지면 브랜드가 아니라 "규칙만 지킨 화면"이 나온다.
- **누가 문서로 정리하는가**: design-prompter가 위 관찰 내용을 받아 `docs/design/confirmed/{작업명 슬러그}.md`(design-prompter가 작업명을 영문 kebab-case로 슬러그화)에 정리한다. design-prompter는 Figma 도구가 없으므로 직접 보지 않고, 위에서 관찰한 내용을 발전시켜 문서화하는 역할이다(디자인 프롬프터의 기존 "브리프 발전" 역할과 동일한 성격 — 원본을 임의로 바꾸지 않고 실행 가능하게 구체화).
- **파일 갱신 방식**: 새 라운드가 확정될 때마다 같은 파일에 새 섹션으로 추가한다(1차 확정 섹션, 2차 확정 섹션 등) — 덮어쓰지 않는다. 이 문서는 로그가 아니라 살아있는 근거 자료라서 5개 캡 규칙(agent-memory)을 적용하지 않는다.
- **후속 워커의 의무**: design-systems, ui-designer 등 이후 단계 워커는 작업 시작 시 이 확정 스펙 문서를 반드시 먼저 읽는다 — 자기 메모리 요약이나 자기 판단으로 재해석하지 않는다.

### 2-4. 사용자가 직접 만든 확정 디자인 — AI 파일럿보다 우선하는 소스

사용자가 Figma에서 직접 화면을 디자인해 "메인 디자인"으로 삼는 경우가 있다. 이건 design-pl 이하 팀이 만든 파일럿보다 **우선하는 최종 소스**다 — 계속 ui-designer에게 위임하는 대신 사용자가 직접 핸들링하기로 판단한 결과이기 때문이다.

- **절대 건드리지 않는다**: "확정 디자인 - 원본 수정 금지" 같은 보호 라벨이 붙은 프레임은 내용은 물론 페이지 이동조차 사용자 확인 없이 하지 않는다. 다른 작업(예: 페이지 통합) 중 우연히 섞여 있는 걸 발견하면, 삭제·재배치를 임의로 판단하지 말고 **즉시 메인 세션에 보고해서 확인받는다**(실제로 이렇게 대응한 사례가 있었고, 그게 맞는 대응이었다).
- **전용 구역**: 이런 사용자 직접 디자인은 "레퍼런스" 페이지와 같은 성격("팀은 여기 쓰지 않는다")으로 취급하되, 레퍼런스가 무드보드 참고용인 것과 달리 이건 실제 확정된 최종 결과물이라는 차이가 있다. 사용자가 지정한 페이지/이름을 그대로 존중한다.
- **design-systems는 여기서 추출한다**: 사용자의 확정 디자인이 있으면, design-systems는 (파일럿을 거치지 않고) 이 디자인에서 직접 토큰/컴포넌트를 추출한다 — 3-B번의 "확정된 것에서 시스템을 추출한다"는 원칙과 같은 방향이며, 그 확정본이 AI 파일럿이 아니라 사용자 원본이라는 점만 다르다.
- **ux-designer는 같은 라운드에서 흐름·문구를 검토한다(시각 추출과 병행, 생략 불가)**: design-systems·graphic-designer가 토큰/컴포넌트/아이콘이라는 "시각 시스템"을 뽑는 동안, ux-designer는 확정 프레임을 `docs/planning`(화면정의서·PRD)과 대조해 **흐름 논리·문구·네이밍 일관성**을 검토한다 — 이건 시각 추출로는 안 잡히는 별개 결이다. 과거엔 이 검토가 파이프라인에서 빠져 있어서, 화면 간 용어 불일치(예: 로그인 링크는 "비밀번호 찾기"인데 실제 기능은 재설정) 같은 흐름상 오류를 design-systems/graphic-designer가 아니라 사용자가 직접 찾아내야 했다(2026-07-14 사례 이후 이 항목 신설). 발견한 불일치는 design-pl에게 보고해 사용자 확인을 받은 뒤 처리한다(ux-designer는 확정 프레임을 직접 고치지 않는다 — 발견만 한다).
- **추출·정리가 끝나면, design-qa를 부르기 전에 스스로 먼저 원본과 재대조한다(자체 점검, design-qa 감사를 대체하지 않음)**: 추출·등록·문서화 과정에서 실측하지 않고 옮겨 적거나("아마 이 값이겠지"), 비슷한 이름의 기존 토큰에 잘못 재바인딩하는 식으로 값이 틀어지는 사고가 실제로 반복됐다(예: Row Action Button 보더가 `ink/900`에 잘못 흡수됨, Table Row Action 텍스트가 13px로 잘못 등록됨 — 둘 다 원본은 다른 값이었다). 그래서 design-systems는 컴포넌트/토큰 등록을 "완료"로 보고하기 전에, 방금 등록한 항목 각각을 확정 프레임의 원본과 `get_screenshot`/`get_design_context`로 다시 대조한다(요약 메모리 재사용 금지, 매번 실측) — 색상 hex, 크기·spacing, 텍스트 스타일(폰트/크기/굵기), 보더/그림자 바인딩이 원본과 정확히 일치하는지 항목 단위로 확인하고, 불일치를 발견하면 design-qa를 기다리지 않고 그 자리에서 바로 고친다. 이건 design-qa의 사후 감사를 생략해도 된다는 뜻이 아니다 — design-qa는 여전히 독립적으로 다시 감사한다(2인 검증), 다만 design-systems 스스로도 "일단 만들고 감사에서 걸리면 고치자"는 태도로 넘기지 않는다.
- **추출은 매번 커버리지 감사와 세트다(기본 동작, 별도 요청 불필요)**: 사용자가 "확정 디자인에서 추출해줘"라고만 말해도, design-pl은 추출 작업에 이어서 그 결과가 `docs/planning`(화면정의서/PRD)이 요구하는 범위를 다 커버하는지 자동으로 함께 확인한다.
  - **빠진 아이콘**: graphic-designer가 이미 확정된 스타일 언어(기존 아이콘들의 strokeWeight/Basic·Visual 트랙 구분/색상 규칙/조형 관례)를 그대로 따라 새로 그려서 보강하고 `docs/design/graphic-assets.md`에 같은 형식으로 추가한다 — 확정 프레임에 없는 걸 임의로 지어내는 게 아니라, 이미 확정된 스타일의 자연스러운 연장이어야 한다.
  - **빠진 표준 폼 컨트롤**: Button/Input/Select처럼 이미 챙기고 있는 것 외에도, **Checkbox/Radio/Toggle(스위치)** 같은 표준 폼 컨트롤은 컴포넌트 시스템의 기본 구성 요소다. 커버리지 감사 때 확정 프레임 전체를 다시 훑어서, 이런 폼 컨트롤이 화면 안에 **원본 도형(raw shape)으로만 존재하고 아직 정식 컴포넌트로 등록되지 않은 곳이 있는지** 확인한다(실제 사례: 로그인 화면의 "☐ 로그인 상태 유지" 체크박스가 컴포넌트화 없이 방치돼 있었다, 2026-07-14 발견). 발견하면 그 확정 프레임에 실제로 쓰인 형태·크기·색을 그대로 정식 컴포넌트로 등록한다(Checked/Unchecked, 필요하면 2-5번 기본 상태 체크리스트에 준해 Focus/Disabled까지) — 화면에 안 쓰이고 있어도 이 프로젝트 성격상(폼이 있는 서비스) 필요할 가능성이 높은 컨트롤(Checkbox/Radio/Toggle)은 커버리지 감사 때 존재 여부를 항상 점검 항목에 포함한다.
  - **빠진 화면/플로우**: 지금 당장 다 만들지 않는다. 목록으로 정리해 메인 세션에 보고만 한다(`docs/design/missing-screens.md` 등) — 실제 제작은 3-B번 "전체 확장" 단계에서, **그리고 3-B번 4-1번의 "디자인 시스템 최종 승인" 게이트를 통과한 뒤에만** 진행한다. 이 경로(사용자 확정 디자인에서 추출)는 "파일럿 승인" 단계 자체가 없으므로, design-systems의 추출·감사·인터랙션 상태 신설이 다 끝난 시점에 4-1번 게이트가 적용된다 — 커버리지 감사가 끝났다고 곧바로 누락 화면 제작으로 넘어가지 않는다.
  - **레거시 정리**: 이전 라운드(AI 파일럿 등)에서 이미 만들어진 토큰/컴포넌트/화면이 있다면, 삭제하지 않고(house rule) `[Legacy ...]` 접두사나 `❌ 미채택 — ` 라벨로 새 확정 산출물과 명확히 구분해, 소비하는 워커가 헷갈리지 않게 한다. 문서(`docs/design/design-system.md` 등) 맨 위에 "신규 산출물만 정식 소스"라고 명시한다.
  - **레거시 컴포넌트를 Assets/Insert 패널에서 없앨 때(COMPONENT/COMPONENT_SET→FRAME 전환)는 순서가 중요하다 — 인스턴스가 있으면 먼저 분리(Detach Instance)하고 나서 전환한다**: 컴포넌트 정의를 FRAME으로 바꾸는 시점에 그 컴포넌트를 참조하는 인스턴스가 아직 남아있으면, 그 인스턴스들은 원본을 잃고 빈 박스로 깨진다(실제로 이 순서를 건너뛰어 인스턴스가 깨진 사례가 있었다). 안전한 순서는: ① 해당 컴포넌트의 모든 인스턴스를 찾는다 → ② 각 인스턴스에 Figma의 "인스턴스 분리(Detach Instance)"를 적용한다(현재 보이는 모습 그대로 독립 레이어로 바뀌고 시각적 손상이 없다, 분리 전/후 스크린샷으로 대조 확인) → ③ 인스턴스가 0개가 된 걸 확인한 뒤에만 컴포넌트 자체를 FRAME으로 전환한다. "인스턴스가 있으니 해제를 보류한다"가 기본 대응이 아니다 — 분리부터 하면 인스턴스 존재 여부와 무관하게 안전하게 해제할 수 있다.
- **ui-designer는 확장만 한다**: 다른 화면으로 패턴을 넓힐 때 사용자의 확정 디자인을 그대로 따르고, 그걸 다시 만들거나 "개선"하려 하지 않는다 — 사용자가 이미 최종 판단을 내린 부분이다.

### 2-5. 컴포넌트는 만들고 끝이 아니다 — 클릭해야만 보이는 상태로 남기지 않고, 한눈에 보이는 문서화 페이지를 항상 같이 만든다

design-systems가 컴포넌트에 Default/Hover/Press/Focus/Disabled/Loading 같은 variant를 다 채우고 레이어 이름·컴포넌트 이름·description까지 꼼꼼히 정리해도, Figma 캔버스에서 그 컴포넌트를 클릭해 variant 피커를 열어야만 각 상태가 보인다면 사람이 한눈에 검토할 수 없다 — "코드/토큰으로 추출 가능한 상태"와 "사람이 훑어보고 확인 가능한 상태"는 다른 문제다. 실제로 이 문제가 있었다(이 프로젝트의 Button 컴포넌트 — 레이어·컴포넌트 이름은 다 잘 정리돼 있었지만, 클릭하지 않고는 Default/Hover/Press/Focus/Disabled/Loading 상태를 볼 수 없었다. 구체적 노드 ID는 프로젝트마다 달라지므로 여기 적지 않는다).

**그래서 컴포넌트/토큰을 새로 만들거나, 기존 컴포넌트에 새 상태(variant)를 추가할 때마다, 그 옆(또는 전용 문서화 구역)에 다음을 갖춘 "스펙 시트" 프레임을 항상 함께 만든다:**
- **제목**: 컴포넌트 이름(예: "Button").
- **설명**: 이 컴포넌트가 뭘 위한 건지, 어디에 쓰이는지 한두 줄(예: "모달 저장/삭제, 인증 카드 CTA 공용").
- **모든 variant를 캔버스에 실제로 펼쳐놓은 그리드**: Default/Hover/Press/Focus/Disabled/Loading 등 상태별 인스턴스를 나열하고, 각 인스턴스 아래에 그 상태 이름을 텍스트 라벨로 붙인다 — variant 피커를 열지 않아도 스크롤/줌만으로 전체 상태를 훑어볼 수 있어야 한다.

이 스펙 시트는 FOUNDATIONS 구역 안에 컴포넌트별로 정리된 전용 구역(페이지 또는 섹션)에 모은다. **담당은 design-systems**(컴포넌트/토큰을 만드는 주체) — 컴포넌트 등록 작업의 일부로 취급하며, "컴포넌트 등록 완료"는 변수/컴포넌트뿐 아니라 이 스펙 시트까지 갖춰졌을 때를 기준으로 한다. interaction-designer가 새 State 축(Hover/Press/Focus/Disabled/Loading 등)을 추가한 직후에는, design-systems가 그 컴포넌트의 스펙 시트를 갱신한다(둘 다 새로 만드는 게 아니라 기존 시트에 새 상태 열/행을 추가).

**같은 원리가 토큰(컴포넌트가 아니라 색상/타이포/spacing/elevation)에도 적용된다**: FOUNDATIONS 구역의 "Colors"/"Typography"/"Spacing"/"Elevation" 페이지에는 이미 스와치 카탈로그 프레임(예: "Colors — Design Tokens" — Primitives 행 + Semantic 행)이 있다. 새 토큰을 만들 때마다 `docs/design/design-system.md`에 텍스트로 기록하는 것과 별개로, **이 스와치 카탈로그에도 그 자리에서 스와치를 추가한다** — 나중에 몰아서 하지 않는다. 실제로 이 프로젝트에서 카테고리 색·에러 색·뮤트 텍스트 색 등 다수의 토큰이 문서(md)에는 기록됐지만 이 시각 카탈로그는 갱신되지 않은 채 방치됐다(2026-07-15 사용자 발견) — "토큰 등록 완료"의 기준도 컴포넌트와 동일하게, md 기록 + 이 카탈로그 갱신 둘 다를 포함한다.

## 3. 시안 단계 화면 vs 정식 SCREENS 단계 화면 — 해상도/완성도 기준

- **시안(Concept) 단계 화면**(2-1번, 또는 화면 레이아웃 자체를 처음 정할 때의 시안 3개): 사이즈 제약 없음, 톤/레이아웃 방향 비교가 목적인 경량 목업. 프로덕션 품질일 필요 없다.
- **정식 SCREENS 단계 화면**(확정 이후 ui-designer가 만드는 최종 화면): 이 프로젝트는 PC 브라우저 대상이므로, 실제 데스크톱 웹 해상도(폭 1440px 기준을 기본값으로 하되, `docs/planning` 문서에 명시된 값이 있으면 그것을 우선한다)로 제작한다. `docs/planning`(화면정의서/기능정의서/구현요구사항/PRD)에 정의된 요소·필드·문구를 실제로 반영해 완성도 있게 만든다 — 시안 단계의 축소 목업을 그대로 확대하지 않고 다시 제작한다.

### 3-B. 정식 SCREENS 단계 진행 순서 — 파일럿(디자인 먼저) → 시스템 추출 → 피드백 → 전체 확장

컨셉(1차)과 레이아웃(2차)이 모두 확정되면, docs/planning에 정의된 화면 전체를 한 번에 만들지 않는다. **디자인 시스템을 먼저 만들고 거기 맞춰 화면을 조립하지 않는다 — 확정된 디자인을 먼저 완성도 있게 발전시키고, 그 결과에서 시스템을 추출한다.** 순서가 반대로 가면(시스템을 먼저 독립적으로 설계) 파일럿이 확정 시안과 다른 것으로 재해석되어 나오는 문제가 생긴다(실제 발생 사례). 다음 순서로 진행한다:

1. **파일럿 화면 선정 (기본 3개)**: design-pl이 docs/planning에서 전체 화면 목록을 파악하고, 그중 서비스의 핵심 플로우를 대표하는 화면 3개를 파일럿으로 선정한다(예: 로그인, 메인 목록, 입력 폼 등 — 구체적으로 어떤 화면인지는 프로젝트마다 다르므로 고정하지 않는다).
2. **파일럿 = 확정된 2차 시안의 발전(elevate), 재설계가 아니다**: ui-designer는 확정된 2차 시안(레이아웃 구조, 컴포넌트 배치, 톤)을 그대로 이어받아, 2-3번의 확정 스펙 문서와 Brand Guide를 근거로 3번 항목 기준(프로덕션 해상도, docs/planning 요소 반영, 상태·디테일까지)의 완벽한 완성도로 발전시킨다. design-systems 토큰은 아직 없으므로(다음 단계에서 이 파일럿을 근거로 만들어짐) 확정 스펙 문서의 원본 값을 직접 쓴다. 아이콘이 필요하면 graphic-designer에게 그 자리에서 요청해 받아 쓴다.
3. **피드백 루프**: 메인 세션이 파일럿 화면을 사용자에게 보여주고, 사용자가 수정을 요청하면 담당 워커가 반영해 다시 보여주는 과정을 승인될 때까지 반복한다. 이건 새 확정 게이트가 아니라 같은 화면을 계속 다듬는 반복 단계다 — 매 수정마다 시안 3개를 새로 만들지 않는다.
   - **이 단계에서는 design-systems를 부르지 않는다.** 아직 파일럿이 최종 승인되지 않았으므로, 결함이 발견돼도 design-systems가 토큰/컴포넌트를 고치러 오지 않는다 — ui-designer가 파일럿 화면에 직접(확정 스펙 문서의 원본 값, 또는 기존 인스턴스의 로컬 오버라이드/detach로) 반영한다. design-systems는 사용자가 파일럿을 최종 승인한 뒤 4번에서 **딱 한 번** 호출된다. 실제로 이 순서가 거꾸로 지켜지지 않아 승인 전에 컴포넌트를 여러 차례 고치는 시행착오가 있었다 — 반복하지 않는다.
   - **이 단계의 반복본도 덮어쓰지 않는다 — 순서가 중요하다.** 수정을 시작하기 **전에 먼저** 기존 파일럿 프레임 이름에 `❌ 폐기 —`(사유 포함) 라벨을 붙인다(이 시점엔 내용 무수정). **그다음** 그 프레임을 복제하거나 새로 만들어 옆에 배치하고, 그 새 프레임 위에서만 수정 작업을 한다. 기존 프레임의 자식 인스턴스를 detach하거나 속성을 그 자리에서 직접 바꾸는 방식("그 자리에서 고치기")은 하지 않는다 — 그렇게 하면 라벨을 나중에 붙이려 해도 "고치기 전" 상태 자체가 이미 사라져 복원이 안 된다(실제로 이 순서를 어겨 이전 상태가 유실된 사례가 있었다). 인스턴스 단위의 작은 수정이라도 예외 없다.
4. **파일럿 승인 후 — design-systems가 시스템을 추출(extract)한다**: 사용자가 파일럿에 만족하면(신뢰 형식), design-systems가 그 승인된 파일럿 화면들에서 실제로 쓰인 값들을 Primitive→Semantic→Component 토큰과 컴포넌트로 추출·등록한다(아이콘도 동일하게 Icons에 등록). 파일럿과 무관하게 새로 설계하지 않는다 — 파일럿에 없던 값을 임의로 추가하지 않는다(파일럿에 없지만 다른 화면에 필요한 값은 그때 가서 파일럿의 패턴을 따라 확장한다).
4-1. **전체 확장 전 별도 게이트 — "디자인 시스템 최종 승인"(프로젝트 규모 무관, 예외 없음)**: 4번(추출)이 끝나고 커버리지 감사·아이콘 보강·인터랙션 상태 신설까지 마쳤어도, **5번(전체 확장)으로 자동으로 넘어가지 않는다.** 지금까지 만들어진 `docs/design/design-system.md`(토큰/컴포넌트/인터랙션 상태 전체)를 사용자가 최종 승인해야만 5번을 시작한다. 이 게이트는 6번의 "100개 넘는 대형 프로젝트만 단계적 승인" 규칙과 별개다 — 화면 수와 무관하게 모든 프로젝트에 적용된다. design-pl은 4번 완료 시점에 반드시 멈추고, 승인 대기 중임을 메인 세션에 보고한다(신뢰할 수 있는 승인 형식 없이는 5번의 워커를 호출하지 않는다). `docs/design/missing-screens.md` 같은 "누락 화면 목록"이 이미 정리돼 있어도, 이 승인 전까지는 목록에 있는 화면을 만들지 않는다.
5. **전체 확장**: 4-1번 승인을 받은 뒤에만, 추출된 시스템을 기준 삼아 docs/planning에 정의된 나머지 화면 전체를 같은 완성도로 제작한다.
6. **대형 프로젝트(전체 화면이 대략 100개를 넘는 경우)는 5번을 한 번에 진행하지 않는다**: 화면 수(3-C번의 상태/플로우 변형 포함)가 대략 100개를 넘는 규모라면, 나머지 화면을 기능 단위의 "큰 카테고리"(예: 인증/온보딩, 메인 대시보드, 결제, 설정 등 — 프로젝트마다 다름)로 나누고, 카테고리 하나를 완성할 때마다 사용자 승인을 받은 뒤 다음 카테고리로 넘어간다. 화면 수가 적은 프로젝트(이 프로젝트 포함)는 이 단계적 승인 없이 5번을 한 번에 진행한다.

### 3-C. 화면 수에는 상태/플로우 변형도 포함된다

같은 화면이라도 실질적으로 레이아웃/구성이 달라지는 상태(빈 목록, 로딩, 에러, 데이터 있음 등)나 플로우 분기(예: 로그인 성공 vs 실패, 카테고리 있음 vs 없음)는 하나의 화면이 아니라 **별도 화면**으로 취급해 디자인한다 — SCREENS 단계 작업 범위·화면 수 산정(3-B번의 100개 기준 포함)에도 포함시킨다. 문구나 색만 바뀌는 사소한 상태(예: 버튼 hover)는 interaction-designer의 몫이라 여기 포함하지 않는다 — 여기서 말하는 건 레이아웃/구성 자체가 달라지는 상태다. docs/planning(화면정의서 등)에 상태별 화면이 명시돼 있으면 그 목록을 그대로 따른다.

## 4. design-prompter가 계획을 세울 때

작업이 "처음 정하는 방향"인지 "이미 확정된 방향의 후속 적용"인지를 계획 단계에서 먼저 판단하고, 전자라면 워커에게 줄 프롬프트에 "시안 3개로 만들어라"를 명시한다. 브랜드/톤앤매너 컨셉이면 2-1번에 따라 brand-designer 브리프와 ui-designer(적용 화면) 브리프를 순서대로 준비하도록 design-pl에게 안내하고, 3개 시안이 서로 다른 트렌드/레이아웃에 대응하도록 브리프에 명시한다. 아이콘·오브제가 필요한 작업이면 2-2번에 따라 담당(brand-designer 방향 → graphic-designer 원화 제작 → design-systems 등록 → ui-designer 조립, 애니메이션이 필요하면 motion-designer)을 브리프에 반영한다. **"후속 적용" 작업의 브리프를 쓸 때는 `docs/design/confirmed/`에 관련 확정 스펙 문서가 있는지 먼저 확인하고, 있으면 그 문서를 워커가 반드시 참고하도록 브리프에 명시한다** (2-3번) — 헤드라인 요약(에이전트 메모리)만 보고 브리프를 쓰면 확정된 디테일이 누락될 수 있다.
