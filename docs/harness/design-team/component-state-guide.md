# 컴포넌트 상태 커버리지 & 네이밍 가이드

이 문서는 design-systems가 컴포넌트를 만들 때 참조하는 **상태(State) 커버리지·네이밍 표준**이자, design-qa가 완성된 컴포넌트를 감사할 때 쓰는 **검증 기준**이다. `docs/harness/design-team/figma-page-format-guide.md`·`icon-craft-guide.md`와 같은 성격의 포터블 참조 문서로, 구체적 hex 값이나 노드 ID를 담지 않고 일반 원칙만 담는다. Focus 링의 정확한 색·spread 값, 프로젝트가 실제로 채택한 토큰 이름 등은 항상 `docs/design/design-system.md`를 참고한다.

**이 문서가 다루지 않는 것**: 애니메이션 duration·easing은 `motion-timing-guide.md`, 토큰 3계층 구조 자체는 `token-architecture-guide.md`, 아이콘 트랙 구분은 `icon-craft-guide.md`를 따른다. 이 문서는 오직 "컴포넌트가 어떤 상태를 갖춰야 하고, 그 상태를 Figma variant로 어떻게 이름 붙일지"만 다룬다.

## 1. State 축 네이밍 규칙

- 상태를 표현하는 variant 속성 이름은 항상 `State`로 통일한다.
- 값은 영문, 대문자 시작으로 통일한다: `Default`/`Hover`/`Press`/`Focus`/`Disabled`/`Loading`(해당 컴포넌트에 의미 있는 것만 사용).
- 기존에 다른 축(Style/Size/Category 등)이 있으면 그 옆에 `State` 축을 추가하는 것이지 대체하지 않는다 — 같은 ComponentSet의 모든 variant는 동일한 속성 키 집합을 가져야 한다.
- **Focus는 별도의 독립 축(예: `Focus=Yes/No`라는 boolean property)으로 만들지 않는다 — 위 `State` 열거형 안의 값 하나로 취급한다.** 별도 축을 만들면 "더 정확해 보이지만" 과설계다. 단, 다른 상태와의 조합(예: 포커스 상태에서 값이 채워진 채 에러가 난 조합)이 실제로 필요하다고 판단되면 그 조합만 `State` 값으로 추가한다 — 무조건 값 하나로 접지 않는다.
- Focus variant의 내용은 항상 대응하는 비-포커스 상태를 그대로 clone한 뒤 포커스 링 효과만 추가해서 만든다 — 배경·보더·텍스트를 독립적으로 새로 그리거나 값을 재입력하지 않는다(재입력은 토큰 바인딩이 끊기는 사고로 이어지기 쉽다).

## 2. 컴포넌트 유형별 필수 상태 체크리스트

컴포넌트를 등록·갱신할 때마다, 확정 디자인에 그 상태가 안 보여도 아래 목록은 예외 없이 전부 갖춘다 — "있으면 만든다"가 아니라 "항상 만든다":

- **버튼류**: Default / Hover / Press / Focus / Disabled (+ 제출성 액션은 Loading 추가)
- **텍스트 입력(Input)**: Default(빈 값) / Placeholder(안내 문구 노출 — Default와 시각적으로 같아 보여도 별도 variant로 문서화) / Focus / Error(유효성 검사 실패 표시) / Disabled
- **선택형(Select/Dropdown)**: Default(닫힘, 값 없음) / Placeholder / Focus / Open(옵션 목록이 펼쳐진 상태 — 펼쳐진 패널의 뼈대만이 아니라 옵션 각각의 hover 강조까지 실제로 디자인) / Error / Disabled
- **체크박스/라디오/토글류**: Default / Checked / Focus / Disabled
- 위 목록에 없는 새 컴포넌트 유형은 "이 컴포넌트가 실무에서 흔히 갖추는 기본 상태가 뭔지"를 스스로 점검해 빠뜨리지 않는다.

**예외 — Focus×Error×Placeholder 3중 조합은 만들지 않는다**: 포커스 중엔 보통 유효성 에러를 지연 표시(blur 후에만 검증)하는 게 일반적 UX 관례라 실사용 빈도가 없다. Focus×Error(값이 채워진 상태에서의 조합)는 계속 만든다 — 제외되는 건 "포커스 중인데 아직 값도 없고 에러도 뜬" 조합뿐이다.

## 3. Disabled 상태는 opacity가 아니라 명시적 색 토큰으로 표현한다

- 컨테이너 opacity를 낮추는 방식은 배경-텍스트 조합에 따라 대비가 들쭉날쭉해지고, 그림자 등 스타일 계층까지 함께 옅어진다.
- 대신 배경/보더/텍스트 3종의 전용 색 토큰을 만들어 opacity는 항상 1로 유지한 채 색 자체로 비활성을 표현한다.
- 상호작용 가능한 컴포넌트 전체(버튼류, 아이콘 버튼, 행 액션 버튼, 체크박스/라디오/토글 등)에 이 3종을 동일하게 통일 적용한다 — 컴포넌트마다 제각각의 disabled 룩을 만들지 않는다.
- WCAG 1.4.11 비텍스트 대비(흰 배경 기준 최소 3:1)를 기본 목표로 계산한다. 사용자가 명시적으로 완화를 요청하면 미만으로 조정할 수 있으나, 그 경우 "명시적 요청으로 기준 미만" 근거를 문서에 남긴다.
- **예외 — 값을 보여주는 표시형 컴포넌트(입력창 등)**: 이미 입력된 값을 계속 읽을 수 있어야 하는 컴포넌트는 배경·보더만 위 공식을 따르고, 텍스트는 Default 상태와 동일한(진한) 색을 유지한다. 이 예외는 표시형 컴포넌트에만 적용하고, 조작형 컴포넌트(버튼 등)에는 적용하지 않는다.

## 4. 아이콘이 포함된 컴포넌트는 항상 등록된 Icon 컴포넌트의 INSTANCE로 조립한다

체크박스의 체크마크처럼 아이콘을 포함하는 컴포넌트를 만들 때, 그 아이콘을 컴포넌트 내부에 raw 벡터로 직접 그리거나 붙여넣지 않는다. 반드시 독립된 Icon 컴포넌트로 먼저 등록(이미 있으면 재사용)하고, 그 INSTANCE를 부품으로 조립한다 — 아이콘이 raw VECTOR로 박혀 있으면 §5의 인스턴스 단위 색상 오버라이드가 자연스럽게 적용되지 않고, 컴포넌트 재사용성도 떨어진다.

## 5. 색상 등 시각 속성은 새 variant가 아니라 인스턴스 단위 오버라이드로 처리한다

색상만 다른 반복 사용 사례(예: 밝은 배경용 컴포넌트를 어두운/강조 배경 위에 쓸 때 텍스트·아이콘 색만 달라지는 경우)는 새 variant나 별도 컴포넌트를 만들지 않는다. 마스터는 가장 흔한/기본 컨텍스트의 값을 유지하고, 예외적인 배치 컨텍스트에서만 그 인스턴스에서 fill/stroke를 로컬 오버라이드한다.

## 체크리스트 (만들 때 / 감사할 때 공통)

- [ ] variant 축 이름이 `State`이고, 값이 영문 대문자 시작 표준값(Default/Hover/Press/Focus/Disabled/Loading)과 일치하는가?
- [ ] Focus가 별도 독립 축이 아니라 `State` 열거형 안의 값으로 들어가 있는가?
- [ ] Focus variant가 대응 상태의 clone + 링 추가로만 만들어졌는가(배경·보더·텍스트가 독립적으로 재입력되지 않았는가)?
- [ ] 이 컴포넌트 유형에 필요한 상태를 위 §2 체크리스트대로 전부 갖췄는가? (Input/Select는 특히 전부 필수)
- [ ] Focus×Error×Placeholder 3중 조합처럼 만들지 않기로 한 조합을 실수로 만들지 않았는가?
- [ ] Disabled가 opacity가 아니라 명시적 색 토큰(배경/보더/텍스트) 3종으로 표현됐는가?
- [ ] 아이콘을 포함한 컴포넌트라면, 그 아이콘이 raw 벡터가 아니라 등록된 Icon 컴포넌트의 INSTANCE인가?
