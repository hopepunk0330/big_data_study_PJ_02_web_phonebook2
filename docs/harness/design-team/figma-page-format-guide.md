# Figma 문서화 페이지 포맷 가이드

이 문서는 FOUNDATIONS 구역(Colors/Typography/Spacing/Elevation/Icons)과 "Component Specs" 페이지처럼, **design-systems가 Figma 안에 만드는 "문서화 페이지"의 시각적 포맷**을 다룬다. `docs/harness/report-format-guide.md`가 doc-writer의 PDF 보고서에 대해 하는 역할과 같다 — 그 문서가 원본 PDF의 디자인을 직접 베끼는 대신 "구조 템플릿"만 재사용하듯, 이 문서도 실제 프로젝트가 채택한 구체적 수치(cornerRadius 몇 px, 스와치 크기 몇 px 등)를 담지 않고 **일반 원칙과 체크리스트**만 담는다. 실제 채택 값은 항상 `docs/design/design-system.md`(프로젝트별 State Ledger)를 참고한다.

**이 문서가 다루지 않는 것**: 어떤 컴포넌트/토큰을 만들지, 어떤 색을 쓸지는 다루지 않는다 — 그건 `docs/harness/design-team/figma-file-organization.md`(페이지 구조·워크플로우)와 각 에이전트 정의(`.claude/agents/design-systems.md` 등, 토큰 3계층·컴포넌트 상태 체크리스트)의 영역이다. 이 문서는 오직 "이미 결정된 토큰/컴포넌트를 문서화 페이지에 어떻게 배치·서식할지"만 다룬다.

## 왜 필요한가

컴포넌트/토큰 자체의 설계(색상 대비, 3계층 구조, 상태 커버리지 등)가 전부 올바르더라도, 그걸 눈으로 훑어보는 문서화 페이지의 서식이 페이지마다 다르면 사람이 신뢰하기 어렵다. 실제로 한 프로젝트에서 컬러 스와치 카탈로그의 일부 칩은 모서리가 둥글고 일부는 각져 있었고, 컴포넌트 스펙 시트도 일부는 정돈돼 있고 일부는 요소를 그냥 붙여넣기만 한 상태였다 — 원인은 토큰/컴포넌트 설계 오류가 아니라, "문서를 어떻게 배치·서식할지"에 대한 명시적 표준이 없었기 때문이었다. 새 토큰/컴포넌트를 추가할 때마다 그 순간의 편의대로 서식을 정하면 이런 표류가 반복된다.

## 1. 스와치(컬러칩) 카탈로그 — Colors/Elevation 등 원시값 나열 페이지

토큰 하나를 시각적으로 보여주는 최소 단위(주로 색상 사각형 + 이름 라벨 + 값 라벨)를 이 문서에서는 "스와치"라고 부른다.

- **같은 카탈로그(예: Colors 페이지의 Primitives 행, Semantic 행) 안의 모든 스와치는 동일한 cornerRadius를 쓴다** — 전부 각지게(0) 하거나 전부 라운드지게(고정값) 하거나 둘 중 하나로 통일한다. 일부는 각지고 일부는 라운드인 혼재를 허용하지 않는다.
- **같은 카탈로그 안의 모든 스와치는 동일한 보더 유무·색·굵기를 쓴다.** 밝은/흰색에 가까운 값(예: 거의 흰색인 파스텔 톤)은 보더가 없으면 배경과 구분이 안 돼 사실상 보이지 않는다 — 카탈로그 전체가 보더를 쓰기로 했다면 이런 밝은 값도 예외 없이 포함한다.
- **같은 카탈로그 안의 모든 스와치는 동일한 크기(width/height)와 라벨 배치(이름 줄 → 값 줄 순서, 폰트 크기, 색상)를 쓴다.** 새 스와치를 추가할 때는 새 스타일을 발명하지 않고 그 카탈로그의 기존 스와치 하나를 그대로 clone한 뒤 색상·라벨 텍스트만 바꾼다 — 이렇게 하면 새 스와치가 자동으로 기존 스타일을 물려받는다.
- **스와치 사이 간격(gap)도 카탈로그 전체에서 하나의 값으로 통일한다.** 8pt 그리드(4 또는 8의 배수) 원칙을 그대로 따른다.
- 새 원시값/시맨틱 토큰을 등록할 때마다, `docs/design/design-system.md`에 텍스트로 기록하는 것과 별개로 이 카탈로그에도 그 자리에서 스와치를 추가한다(나중으로 미루지 않는다 — `figma-file-organization.md` 2-5번 참고).

## 2. 컴포넌트 스펙 시트 — Component Specs 페이지

컴포넌트 하나의 전체 variant를 한눈에 훑어볼 수 있게 모아둔 프레임을 "스펙 시트"라고 부른다(`figma-file-organization.md` 2-5번에서 이미 요구하는 구성 요소를 서식 관점에서 구체화한 것).

모든 스펙 시트는 다음 순서·구성을 동일하게 따른다:

1. **제목** — 컴포넌트 이름. 페이지 내 다른 스펙 시트의 제목과 같은 폰트/크기/굵기를 쓴다.
2. **설명** — 이 컴포넌트가 뭘 위한 건지, 어디에 쓰이는지 1~2줄. 페이지 내 다른 스펙 시트의 설명과 같은 톤(회색조 보조 텍스트 등)을 쓴다.
3. **전체 variant 그리드** — 상태별(Default/Hover/Press/Focus/Disabled 등, 또는 컴포넌트 성격에 맞는 축) 인스턴스를 빠짐없이 나열하고, 각 인스턴스 아래에 그 상태 이름을 텍스트 라벨로 붙인다. variant 축이 2개 이상이면(예: Style × State) 행/열 그리드로 배치하고 행·열 헤더 라벨을 붙인다 — 헤더 라벨의 x/y 좌표가 실제 셀과 어긋나지 않도록 정렬을 맞춘다.
4. **여백** — 프레임 내부 padding과 섹션(제목/설명/그리드) 사이 여백을 페이지 내 다른 스펙 시트와 동일한 값으로 통일한다.

**컨테이너는 항상 `clipsContent=false`로 만든다.** Focus 링처럼 컴포넌트 바운딩박스 밖으로 나가는 효과가 셀 안에 하나라도 있으면 예외 없이 적용한다 — 그리드를 구성하는 모든 계층(Cell/Row/Grid/루트)에 확인한다.

**variant 축이 늘어나면(예: State 하나가 추가되면) 스펙 시트도 그 자리에서 함께 갱신한다.** 컴포넌트 ComponentSet의 실제 variant 개수와 스펙 시트 그리드의 셀 개수가 항상 일치해야 한다 — 나중에 "다음 라운드에 문서화하며 같이 하겠지"로 미루면 실제 variant와 스펙 시트가 어긋난 채 방치되기 쉽다.

**새 컴포넌트의 스펙 시트를 만들 때는 그 페이지의 기존 스펙 시트 하나를 서식 기준으로 삼는다** — 제목/설명 스타일, 그리드 셀 크기, 라벨 배치를 그대로 재사용하고 내용만 바꾼다. 페이지 안에 스펙 시트가 아직 하나도 없다면(그 페이지 최초 등록), 위 1~4번 구성을 새로 세팅하고 이후 스펙 시트가 그 스타일을 기준으로 삼는다.

## 3. 페이지(문서화 구역) 레벨 정렬

같은 카테고리의 문서화 페이지(예: FOUNDATIONS 전체, 또는 Component Specs 페이지 안에 나란히 놓인 스펙 시트들) 안에서는:

- 프레임들의 정렬 기준선(예: 좌측 x좌표)을 통일한다 — 프레임마다 제각각의 x좌표에서 시작하지 않는다.
- 프레임 사이 여백(세로 간격)도 프로젝트가 정한 하나의 값으로 통일한다.
- 프레임 크기가 커져서(예: variant 추가) 다음 프레임과 겹치게 되면, 그 프레임만 늘리고 끝내지 않는다 — 뒤따르는 프레임 전체의 위치를 재확인해 겹침이 없는지 점검한다.

## 4. 새 문서화 프레임을 만들거나 갱신할 때 확인할 체크리스트

- [ ] 이 스와치/스펙 시트가 속한 카탈로그·페이지에 이미 같은 종류의 프레임이 있는가? 있다면 그 프레임을 서식 기준(clone 원본)으로 썼는가 — 처음부터 새로 그리지 않았는가?
- [ ] (스와치) cornerRadius가 같은 카탈로그의 다른 스와치와 동일한가?
- [ ] (스와치) 보더 유무·색·굵기가 같은 카탈로그의 다른 스와치와 동일한가? (밝은/흰색 계열 값도 예외 없이 포함했는가)
- [ ] (스와치) 크기·라벨 배치(이름/값 줄 순서, 폰트)·스와치 간 gap이 동일한가?
- [ ] (스펙 시트) 제목 → 설명 → 전체 variant 그리드(상태별 라벨 포함) → 여백 순서를 갖췄는가?
- [ ] (스펙 시트) ComponentSet의 실제 variant 개수와 그리드 셀 개수가 정확히 일치하는가?
- [ ] (스펙 시트) variant 축이 2개 이상이면 행/열 헤더 라벨이 실제 셀과 정렬이 맞는가?
- [ ] 프레임(스와치 컨테이너·Cell/Row/Grid·스펙 시트 루트) 전부 `clipsContent=false`로 설정했는가?
- [ ] 같은 문서화 구역 안에서 이 프레임의 정렬 기준선·프레임 간 여백이 다른 프레임들과 통일돼 있는가?
- [ ] 프레임 크기 변경으로 다음 프레임과 겹치지 않는지 재확인했는가?
- [ ] 이 프레임에서 실제로 채택한 구체적 값(cornerRadius, 크기, 간격 등)을 `docs/design/design-system.md`에 근거와 함께 기록했는가? (이 가이드 문서 자체에는 프로젝트 고유 수치나 노드 ID를 적지 않는다)
