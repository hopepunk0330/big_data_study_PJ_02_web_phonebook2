## 1. 개요

본 문서는 05(TRD) §10이 정의한 테스트 전략의 골격을 실행 가능한 수준으로 구체화한 후속 문서입니다. TRD §10이 "어떤 도구로, 어떤 파일 구조로 테스트할 것인가"를 정의했다면, 본 문서는 "정확히 무엇을 검증할 것인가"를 케이스 단위 표로 못박습니다. 이후 개발팀 qa-engineer는 본 문서의 표를 그대로 `tests/` 아래 pytest 케이스로 옮깁니다.

**테스트 대상 범위**: 01 문서 FR-01~15, 04 문서 PRD AC-01~18, 02 문서 화면 SCR-001~004 및 SCR-900(공통 메시지/오류 표시 규칙).

작업 중 발견된 로그아웃 무세션 응답에 관한 문서 간 애매함은 위 "변경 이력" 항목에 기록된 대로 이미 해소되어 본 문서 전체에 단일 결론으로만 반영되어 있습니다(§4-1 `TC-API-AUTH-11` 참고).

## 2. 테스트 환경

모든 테스트 케이스에 다음 사전조건이 암묵적으로 적용됩니다(표마다 반복 기재하지 않습니다).

- PostgreSQL(`pg-lab`) 및 FastAPI 서버(`127.0.0.1:8000`)가 기동 중이다.
- 각 테스트는 **독립적으로 실행 가능**해야 한다 — 테스트 시작 시 매번 새 계정을 생성(예: `username=f"user{uuid4().hex[:8]}"`)하고, 다른 테스트가 만든 데이터에 의존하지 않는다(테스트 독립성 원칙).
- API 테스트는 playwright `APIRequestContext`(`request` 픽스처)로, 화면 흐름(E2E) 테스트는 `page` 픽스처로 수행한다(TRD §10-2).
- 모든 실패 케이스는 상태 코드뿐 아니라 **500이 아님**을 함께 확인한다(NFR-01 / AC-17, 암묵적 공통 어서션 — 표에 매 행 반복 기재하지 않음).
