## 1. 서비스 트라이앵글

- **타겟 사용자** (PRD §3 페르소나 그대로): 사용자 A/B — 자기 연락처를 관리하려고 가입한 개인. 여러 명이 같은 서비스를 쓰지만 데이터는 완전히 격리됨("같은 건물, 각자의 잠긴 서랍"). 학습자(개발자) — 이 서비스를 만들며 실무 도구(REST API·DB·인증·화면 연동)의 동작 원리를 배우는 목적도 함께 가짐.
- **핵심 가치**: 1차 과제(콘솔 프로그램)의 세 한계 해소 — ① 그 컴퓨터에서만 쓸 수 있던 것 → 브라우저만 있으면 어디서든, ② 사용자 구분이 없던 것 → 로그인 기반 데이터 격리, ③ 카테고리가 3개로 고정이던 것 → 사용자 정의 카테고리.
- **기능**: PR-01~PR-13(PRD §6, PR-13은 "비밀번호 재설정")으로 구현 — 가입/로그인, 연락처 CRUD+검색, 카테고리 CRUD, 비밀번호 재설정, 견고성(잘못된 입력 처리), 영속화, 데이터 격리.

이 프로젝트는 수업 과제이며 실제 수익모델은 없다. 그래서 여기서 말하는 "핵심 가치 제안"은 매출이 아니라 **학습 목표 달성**(REST API·DB·인증·화면 연동을 실무 수준으로 익히는 것)과 **PRD 인수 기준(AC-01~18) 충족**이다. 새 기능의 우선순위나 포함 여부를 판단할 때는 "이 기능이 이 두 가치 중 어느 쪽에 얼마나 기여하는가"를 근거로 삼는다 — 유행이라서, 그냥 넣고 싶어서가 아니라.

## 2. BM 핵심 요소

- **가치 제안**: "내 사람들을 어디서든, 나만 볼 수 있게, 내 방식대로 분류해서 관리한다." `docs/design/brand-guide.md`의 포지셔닝("차가운 CRM형 유틸리티가 아니라 사용자가 직접 손으로 정리한 '내 사람들 목록' 같은 정서적 온기 + 로그인 사용자별 데이터 격리라는 신뢰 요소")과 일치하는 표현이며, 화면정의서·브랜드 문서 양쪽에서 같은 컨셉을 공유한다.
- **핵심 리소스**: FastAPI 백엔드, PostgreSQL(사용자별로 격리된 연락처/카테고리 테이블), 세션 기반 인증(쿠키). JWT나 이메일 발송(SMTP) 같은 무거운 리소스는 의도적으로 배제하고 PRD §11 "향후 확장"으로 미뤄 뒀다 — 이번 과제 범위(로컬 개발 환경, 학습자 단독 검증)에 맞춘 리소스 최소화. 비밀번호 재설정(PR-13)도 이 원칙을 그대로 따라 SMTP 없이 기존 세션/인증 스택만으로 구현한다.
- **범위(스코프) 원칙**: PRD §2-2 비목표(N1~N5)가 1차 방어선이다. 새 기능 요청이 들어오면 "① PRD 목표(G1~G6) 중 무엇에 기여하는가, ② 비목표와 부딪히는가, ③ 부딪힌다면 어떤 트레이드오프(보안 vs 개발 범위, 리소스 vs 학습 목적 등)를 감수하는가"를 먼저 판단하고 근거를 남긴다. 최종 포함 여부는 사용자가 결정한다.

## 3. 스코프 결정 — SCR-004 "비밀번호 재설정" (canonical)

> **명칭 안내(2026-07-14)**: 이 화면·기능은 원래 "비밀번호 찾기"라고 불렀으나, 2026-07-14에 "비밀번호 재설정"으로 명칭이 통일되었다(결정·적용 범위는 §3-3 참고). 아래 본문(§3)은 현재 명칭으로 갱신했다. 다만 §3-1·§3-2는 각 라운드 당시 실제로 문서에 반영했던 문구를 그대로 기록해 둔 **감사 기록(audit log)** 이므로, 그 안의 "비밀번호 찾기" 표기는 의도적으로 고치지 않았다 — 현재 유효한 최종 문구는 각 산출물 문서의 최신 버전과 §3-3을 따른다.

- **원본 PRD 상태(v1.0)**: N1 "비밀번호 찾기/변경, 이메일 인증"은 명시적 비목표였다(이유: "인증의 '핵심 뼈대'에 집중 — 확장은 §11"). §11 "향후 확장" 표는 이 기능을 재도입할 때 "이메일 발송, 임시 토큰" 방식을 전제로 적어 뒀다.
- **결정 (2026-07-14, 화면정의서 v1.2)**: 확정 디자인(Figma, 사용자가 직접 만들어 확정한 원본)의 로그인 화면(SCR-001)에 "비밀번호 재설정" 링크가 이미 존재했다(결정 당시 명칭은 "비밀번호 찾기"였다 — 명칭 통일 경위는 §3-3). 디자인이 이미 이 기능을 전제하고 있는 상태에서 PRD의 N1만 그대로 유지한 채 디자인 요소를 무시하기보다, **기능 범위에 포함하기로 결정**했다 — "레퍼런스(확정 디자인)에 이미 반영된 요소를 그대로 베끼지 않고, 이 프로젝트 성격에 맞게 종합해서 포함/배제 근거를 남긴다"는 원칙의 적용 사례.
- **본인 확인 방식 3안 검토**:
  1. 가입 시 보안질문 추가 — 질문 등록 UI·정답 검증 로직이 추가로 필요, 이번 범위에는 과함.
  2. 이메일 발송(토큰 링크) — SMTP 서버가 필요, PRD §11이 원래 전제한 방식이지만 이번 과제 인프라 범위를 넘음.
  3. **아이디만 확인 후 즉시 재설정 (채택)** — 추가 인프라 없이 기존 인증 스택만으로 구현 가능.
- **채택 이유**: 이번 과제의 핵심 평가 기준은 "멀티유저 인증"(로그인·세션·데이터 격리)이지 "본인 확인의 보안 수준"이 아니다. SMTP·보안질문 없이도 "아이디 확인 → 2단계 전환 → 비밀번호 변경"이라는 인증 플로우 자체(2단계 상태 전환, API 연동, 클라이언트 검증)는 그대로 학습·구현할 수 있어, 리소스 대비 학습 가치가 가장 높다고 판단했다.
- **알려진 트레이드오프(의도적 단순화)**: 아이디 존재 여부만으로 바로 재설정을 허용하므로, **아이디만 알면 누구나 그 계정 비밀번호를 바꿀 수 있다.** 실제 서비스라면 부적합한 수준의 보안이지만, 이번 과제 범위에서는 "보안 vs 개발 범위"의 트레이드오프로 허용하기로 결정됐다 — 이 프로젝트에서 앞서 있었던 "아이디만 확인 후 재설정" 유형의 다른 단순화 결정과 동일한 성격이다.
- **AC-04와의 보안 원칙 차이 — 결론(2026-07-14)**: PRD AC-04는 "틀린 비밀번호로 로그인하면 실패 안내가 뜨고, **아이디 존재 여부는 알 수 없는 문구**"여야 한다고 규정한다(사용자 열거 공격 방지 원칙). 반면 SCR-004 1단계는 "존재하지 않는 아이디입니다"라는 404 메시지로 **아이디 존재 여부를 명시적으로 노출**한다. **AC-04를 지우거나 완화하지 않고, 적용 범위를 "로그인 실패 응답(`POST /auth/login`)에 한정"하는 문구로 명시하는 쪽으로 확정**했다 — 비밀번호 재설정 흐름(AC-18)은 애초에 "아이디만 알면 재설정 가능"을 전제로 하므로 아이디 존재 여부를 감출 실익이 없고, 오히려 실패 사유를 알려주는 편이 UX상 낫다는 판단이 유지된다. 정확한 반영 문구는 **§3-1**에 명시.

## 3-1. PRD v1.0 → v1.1 개정 확정 사항 (2026-07-14, planning-writer 반영 완료 — PRD v1.1에 이미 반영됨)

**원문(PRD v1.0 §9)**: `AC-04: 틀린 비밀번호로 로그인하면 실패 안내가 뜨고, 아이디 존재 여부는 알 수 없는 문구다`

### (5) §7 전체 워크플로우 다이어그램 / §4 제품 구성 — 원래 결정 (이력 보존)

(실제로 PRD v1.1에서는 §4/§7 모두 Mermaid로 재작성되었으나, 위 방침에 따라 비밀번호 찾기 흐름 없이 원본 이미지 내용만 그대로 옮겼다 — PRD v1.1 §4/§7 하단 "다이어그램 변환 안내(v1.1)" 블록 참고.)

## 3-2. PRD v1.1 → v1.2 개정 브리프 — §4/§7 사용자 여정 다이어그램에 비밀번호 찾기 흐름 반영 (2026-07-14)

> **명칭 통일 후속 안내(2026-07-14, §3-3)**: 아래 (1)~(4)에 인용된 "비밀번호 찾기"라는 표기(PW 노드 텍스트, changelog 인용 블록, SCR-001 Mermaid 인용 등)는 모두 이 §3-2 라운드 **당시** 실제로 PRD/화면정의서에 반영했던 문구를 그대로 기록한 것이다. 이후 명칭 통일 결정(§3-3)에 따라 PRD/화면정의서의 실제 문구는 "비밀번호 재설정"으로 다시 갱신되지만, 이 §3-2 자체는 감사 기록(audit log) 성격이라 고치지 않는다.

**배경**: 사용자가 §3-1 (5)의 결정("§4/§7 다이어그램에 비밀번호 찾기 흐름을 넣지 않는다")을 명시적으로 뒤집고, PRD §4/§7 Mermaid 다이어그램에 비밀번호 찾기 흐름을 추가해달라고 요청·승인했다. 방향도 이미 사용자가 정했다: "§4(정적 구조도)는 억지로 안 넣거나 최소한의 각주만, §7(순차 여정)에 U2 지점에서 분기하는 요약 노드로 추가하고 SCR-004로 상세를 위임." 아래는 그 방향을 실행 가능한 수준으로 구체화한 것이다.

### (1) §4 "제품 구성" — 최종 권고: 다이어그램(mermaid 코드) 자체는 수정하지 않는다. 본문에 한 줄 각주만 추가한다

**근거**: §4의 flowchart는 "이미 로그인한 A/B가 서비스를 통해 각자의 서랍에만 접근한다"는 **정적 구조**(데이터 격리)를 보여주는 게 유일한 목적이다. A/B 노드 자체가 "로그인 상태로 등장"한다고 명시되어 있어, 로그인이라는 단계 자체가 이 다이어그램의 관심사가 아니다. 비밀번호 찾기는 "로그인 이전, 로그인이 막혔을 때"의 복구 경로이므로 이 정적 구조도에 노드/화살표로 넣으면 "요청마다 누구인지 확인 → 서랍 접근"이라는 단일 메시지가 흐려진다(Simplicity First: 그림 하나에 목적 하나). 반대로 완전히 언급을 안 하면 "PRD 어디에도 비밀번호 찾기 복구 경로가 안 보인다"는 인상을 줄 수 있으므로, 최소 각주로 존재만 알리고 §7로 위임한다.

**추가할 위치·문구**: §4 본문에서 "**주의 깊게 볼 것**: ... B의 '친구'는 그대로입니다." 문단과 "> **다이어그램 변환 안내(v1.1)**" 블록 사이에 아래 한 줄을 새 문단으로 추가한다(mermaid 코드 블록 안이 아니라 코드 블록 바깥의 일반 본문):

> ※ 로그인 실패·비밀번호 분실 시의 복구 경로는 이 구조도의 범위가 아닙니다(이 그림은 로그인 이후의 데이터 격리만 보여줍니다) — 순서 흐름은 §7을 참고하세요.

이 문장 하나만 추가하고, mermaid `flowchart LR` 코드 자체(노드 A/B/S/DA/DB, 화살표)는 한 글자도 바꾸지 않는다.

### (2) §7 "전체 워크플로우" — U2에서 분기하는 요약 노드 1개 + 화살표 2개 추가

**정확한 삽입 지점**: 기존 노드 선언부에 `PW` 노드 1개를 U2 다음 줄에 추가하고, 기존 `V --> U1 --> U2 --> M` 줄 바로 아래에 화살표 2줄을 추가한다. `classDef step` 적용 대상에 `PW`를 포함시킨다(기존 V/U1/U2/LO/R와 같은 스타일 — 새 클래스를 만들지 않음, Simplicity First).

**노드 텍스트(그대로 사용)**:
```
PW["비밀번호 찾기 (UC-09)<br/>아이디 확인 → 새 비밀번호 설정<br/>상세: 화면정의서 SCR-004 참고"]
```

**화살표(레이블 포함, 그대로 사용)**:
```
U2 -->|"비밀번호를 잊음"| PW
PW -->|"재설정 완료 → 로그인 화면 복귀"| U2
```

**합류 지점 판단(U2로 재합류, M으로 직행하지 않음)**: 화면정의서 SCR-004는 비밀번호 재설정 성공 시 "SCR-001(로그인 화면)로 복귀"한다고 규정한다(SCR-004 §4 구성요소-동작 표, Mermaid 흐름의 `J["비밀번호가 변경되었습니다 안내"] --> K[["SCR-001 로그인 화면으로 전환"]]`). PRD §7의 U2 노드가 바로 이 "로그인(로그인 폼이 있는 지점)"을 대표하므로, 논리적으로 같은 지점은 U2다. M(관리 화면)으로 직행시키면 "재설정 후 로그인 절차 없이 바로 관리 화면에 진입한다"는 잘못된 인상을 주므로 채택하지 않는다.

**실패 케이스(존재하지 않는 아이디)는 그리지 않음**: SCR-004의 전체 상태 분기(404/422, 클라이언트 사전검증 등)를 이 요약 다이어그램에 복제하지 않는다. §7은 "여정의 큰 흐름"을 보여주는 문서이고, 실패·재시도 같은 세부 상태는 화면정의서(SCR-004)의 몫이다 — 이 원칙은 PRD의 다른 부분(예: §8 카테고리 삭제 분기)에서도 "정상 흐름 + 대표 예외 1개" 이상으로 세분화하지 않는 것과 같은 결이다. 따라서 이번에 추가하는 디테일은 **노드 1개(PW) + 화살표 2개(U2→PW, PW→U2)**로 확정한다 — 이보다 적으면(각주만) 방향 자체가 사용자 요청과 어긋나고, 이보다 많으면(404/422까지) SCR-004를 복제하는 과잉이다.

**참조 문구**: PW 노드 텍스트 안에 이미 "상세: 화면정의서 SCR-004 참고"를 3번째 줄로 포함시켰으므로, 별도의 화살표 레이블이나 각주로 중복 표기하지 않는다.

**결론**: 이번 변경은 PRD §4/§7의 **다이어그램(시각화) 수준**에서만 일어나며, 기존에 이미 확정된 UC-09/PR-13/AC-18/SCR-004/FR-14/FR-15 중 어떤 것도 새로 만들거나 번호를 바꾸지 않는다(전부 기존 번호를 그대로 인용). 따라서 01/03 문서·tech-architecture.md에는 갱신할 내용이 없다. 근거:

## 3-3. 명칭 통일 — "비밀번호 찾기" → "비밀번호 재설정" (2026-07-14, 기획 문서 4종 개정 브리프)

**배경**: SCR-004로 이동하는 흐름 전체(로그인 화면 링크, SCR-004 제목, PRD UC-09/AC-18 등)를 지금까지 "비밀번호 찾기"라고 불러왔다. 사용자가 이를 "비밀번호 재설정"으로 통일하기로 확정했다(범위: 기획 문서 01/02/03/04 전체. Figma는 사용자가 이미 직접 Link 컴포넌트 텍스트를 "비밀번호 재설정"으로 바꿔 반영 완료 — 이 브리프의 대상이 아니다).

### 두 층위 판단

**층위 1 — 흐름 전체를 가리키는 이름**: SCR-004 화면명, 로그인 화면 링크 문구, PRD UC-09/AC-18, 화면 목록·전이도·매핑 표, Mermaid 노드 라벨 등 **사용자가 마주하는 흐름 전체**를 가리키는 모든 곳은 "비밀번호 찾기" → "비밀번호 재설정"으로 **그대로 1:1 치환**한다. 이 흐름은 애초에 "잃어버린 비밀번호를 찾아주는" 기능이 아니라 "아이디 확인 후 새 비밀번호로 재설정하는" 기능이므로(SMTP도, 임시 비밀번호 발급도 없음), "재설정"이 기능을 더 정확히 서술하는 이름이기도 하다 — 단순 리브랜딩이 아니라 정확성 개선이기도 하다.

**층위 2 — 1단계 개별 기능(FR-14/FN-014/스키마)의 서술**: FR-14·FN-014·`FindPasswordIn/Out`은 "그 흐름의 1단계"만을 가리키는데, 이 1단계가 실제로 하는 일은 "비밀번호를 찾아주는" 것도 "재설정하는" 것도 아니고 **"아이디 존재 여부만 확인하는"** 것이다. 따라서 "찾기"를 기계적으로 "재설정"으로 바꾸면(예: FR-14를 "비밀번호 재설정 (아이디 확인)"으로 바꾸면) 오히려 "1단계에서 이미 비밀번호가 재설정된다"는 오해를 줄 수 있어 부정확하다. **1단계는 "아이디 확인"을 주된 이름으로 쓰고, 상위 흐름과의 관계는 괄호로 "(비밀번호 재설정 1단계)"라고 명시하는 패턴으로 통일**한다 — FN-014("아이디 존재 확인 (비밀번호 찾기 1단계)")는 이미 이 패턴에 가장 가까우므로 괄호 안 단어만 바꾸면 되고, FR-14("비밀번호 찾기 (아이디 확인)")는 주-부 순서가 반대이므로 이번 기회에 순서까지 바로잡는다. FR-15/FN-015(실제 비밀번호 변경, 2단계)는 원래도 "재설정"으로 정확히 서술되어 있었으므로 그대로 두되, FN-015 소제목의 "비밀번호 찾기 2단계"라는 병기만 정리한다(아래 03 문서 항목 참고).

**API 경로·스키마 클래스명은 이 브리프의 범위 밖**: `POST /auth/find-password`, `FindPasswordIn`, `FindPasswordOut` 같은 코드 식별자는 API 계약/구현 수준(tech-architect·개발 소관)이라 이번 기획 문서 개정 브리프에서 변경하지 않는다. 코드 블록(백틱)으로 표기된 리터럴은 전부 그대로 두고, 그 옆의 한국어 설명 문구만 바꾼다.

**원문 인용은 손대지 않는다**: 각 문서가 옛 버전(v1.0 PDF 등)의 문구를 "원문"이라고 명시해 그대로 인용한 블록(예: 이 문서 §3-1 (2)의 "N1 | 비밀번호 찾기/변경, 이메일 인증")은 역사적 사실이므로 고치지 않는다. 각 문서 최상단 "변경 이력" 표의 과거 버전 로그, 그리고 §3-1/§3-2처럼 "그 라운드 당시 실제로 반영한 문구"를 감사 기록으로 남긴 블록도 동일하게 보존한다 — 새 버전 로그 한 줄만 추가한다.

### 크로스 레퍼런스 동기화 주의사항

버전을 4개 문서 모두 동시에 올리므로, 서로를 파일명(버전 번호 포함)으로 인용하는 지점이 스테일해지지 않도록 편집 순서에 주의한다 — 위 01-(7), 02-(7)에서 구체적으로 짚었다. 이 외에도 각 문서를 열 때 "v1.1", "v1.4" 같은 버전 문자열로 다른 문서를 인용하는 곳이 더 있는지 한 번 더 grep으로 확인할 것을 권한다(예: `docs/planning/tech-architecture.md`도 01/03 문서를 버전 번호로 인용하고 있을 수 있으나, 이번 브리프는 기획 문서 4종으로 범위를 한정했으므로 tech-architecture.md의 갱신 여부는 planning-pl이 별도 판단).

## 4. 01/03 문서와의 잔여 불일치 (해결 상태)

PRD v1.0→v1.1 라운드로 PRD 쪽 텍스트 절(§2-2, §5, §6, §9, §11)의 정합화는 처리되었고, 01/03 문서(v1.1)·화면정의서(v1.4)도 이후 라운드에서 FR-14/FR-15 계약 반영 및 "FR-13" 임시 표기 정정까지 완료되어 더 이상 미해결 항목이 없다(2026-07-14 시점 기준). §3-2 라운드(PRD §4/§7 다이어그램)도 §3-2 (4)에서 확인했듯 새로운 불일치를 만들지 않았다. §3-3 라운드(명칭 통일)는 순수 표기 통일이라 새로운 스코프/정합성 이슈를 만들지 않으며, §3-3 "문서별 변경 목록"이 곧 이번 라운드의 실행 브리프다.

PRD v1.0 원문을 직접 재확인해 (1) AC-04를 "로그인 실패 응답에 한정"으로 범위 명시(삭제·완화 아님), (2) N1을 분리해 "이메일 인증"만 비목표로 남기고 "비밀번호 찾기/변경"은 제거, (3) §11 표의 "비밀번호 변경/찾기" 행을 "본인 확인 강화(향후 확장)"로 갱신, (4) UC-09/PR-13/AC-18 신설 문구를 확정. 번호 확인 결과 PRD 원문에 UC-08·PR-12·AC-17까지만 쓰이고 있어 UC-09/PR-13/AC-18이 실제로 빈 번호임을 확인했다. 또한 PRD가 "FR-" 번호를 인용하는 관례가 원문에 없음을 확인해, FR-14/15를 PRD에 직접 인용하지 않고 "01 문서 참조"로만 남기도록 권고했다. §4(구 미해결 절)를 해결 상태로 갱신하고, 화면정의서의 "FR-13" 임시 표기도 PRD 확정 후 후속 라운드에서 정정이 필요함을 planning-pl에게 남겼다.
- **2026-07-14 — PRD v1.1 → v1.2 개정 브리프 (§4/§7 사용자 여정 다이어그램에 비밀번호 찾기 흐름 반영)**: 사용자가 §3-1 (5)의 "다이어그램에 넣지 않는다"는 결정을 명시적으로 뒤집고 승인함에 따라 새 브리프 작성. - **2026-07-14 — 명칭 통일 브리프: "비밀번호 찾기" → "비밀번호 재설정" (이번 라운드)**: planning-pl 요청으로, SCR-004로 이동하는 흐름 전체의 명칭을 "비밀번호 찾기"에서 "비밀번호 재설정"으로 통일하는 개정 브리프 작성(Figma는 사용자가 이미 반영 완료, 이번 브리프는 기획 문서 01/02/03/04 전용). 01/02/03/04 문서 원문을 직접 재확인해 "비밀번호 찾기" 문자열이 나오는 모든 위치를 확인했다. 두 층위로 판단을 나눴다: (층위 1) SCR-004 화면명·로그인 링크·UC-09·AC-18·화면 목록/전이도/매핑 표·Mermaid 노드 라벨 등 **흐름 전체를 가리키는 표기**는 전부 "비밀번호 재설정"으로 1:1 치환(단순 리브랜딩이 아니라 실제 기능 서술의 정확성도 개선 — 이 흐름은 애초에 "비밀번호를 찾아주는" 게 아니라 "재설정하는" 기능이었음). (층위 2) FR-14/FN-014(1단계, 아이디 존재 확인만 함) 같은 **1단계 개별 기능 서술**은 "찾기"를 "재설정"으로 기계 치환하지 않고 "아이디 확인 (비밀번호 재설정 1단계)" 패턴으로 재구성 — 1단계에서 아직 비밀번호가 재설정되는 것은 아니므로 정확성을 위해서다. FN-015 소제목의 "비밀번호 재설정 — 비밀번호 찾기 2단계"라는 기존 병기도 이번에 "비밀번호 재설정 2단계 — 새 비밀번호 저장"으로 정리하도록 브리프에 포함했다. API 경로(`POST /auth/find-password`)·스키마 클래스명(`FindPasswordIn/Out`)은 코드 식별자라 이번 브리프 범위 밖으로 명시(tech-architect 소관). 각 문서의 "원문" 인용 블록과 과거 버전 changelog 로그, 그리고 이 문서 §3-1/§3-2의 감사 기록 블록은 역사적 사실 보존을 위해 고치지 않기로 결정하고, §3-2 상단에 그 취지를 알리는 안내를 추가했다. 4개 문서 모두 마이너 버전을 1씩 올리도록 권고(01: v1.1→v1.2, 02: v1.4→v1.5, 03: v1.1→v1.2, 04: v1.2→v1.3)하고, 상호 버전 인용(01 §3-5 배경 문단, 02 SCR-004 "반영 완료" 문단)이 스테일해지지 않도록 동기화 지점을 §3-3에 명시했다. §3 헤딩과 본문(현재 상태를 나타내는 살아있는 서술)은 새 명칭으로 직접 갱신하고, §3-1/§3-2(각 라운드의 감사 기록)는 원문 그대로 보존했다.
