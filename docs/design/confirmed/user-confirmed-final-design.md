## 1. 브랜드 트라이앵글 (컨셉 요약)

- **포지셔닝**: 차가운 CRM형 유틸리티가 아니라, 사용자가 직접 손으로 정리한 "내 사람들 목록" 같은 정서적 온기 + 로그인 사용자별 데이터 격리(신뢰 요소)를 시각적 경계로 드러낸다.
- **퍼스낼리티**: "다정하지만 경계는 또렷한 친구" — **네오브루탈리즘(두꺼운 잉크 아웃라인 + 하드 오프셋 그림자) + 8비트 픽셀 컨페티(장식용 미니 픽셀 아이콘 산포)**의 하이브리드로 표현된다. "다정함"은 파스텔이 아닌 채도 있는 원색(틸/코랄/앰버)과 장난스러운 픽셀 장식으로, "경계는 또렷하게"는 2~3px 굵은 잉크 보더와 그림자의 딱 떨어지는 오프셋으로 표현된다.
- 이 이중성은 아래 모든 절(컬러, 그림자, 장식 모티프)에 걸쳐 반복되는 원칙이다 — 어느 한 절만 보고 판단하지 않는다.

| 본문/입력값/placeholder/2·3차 행 텍스트 | Noto Sans KR Regular | 12~14px | 0 | 인풋 텍스트, 전화번호·주소 값 |

**위계**: Display(워드마크, Baloo 2) → Heading(Noto Sans KR Black) → 마이크로 라벨(Inter Black, 대문자+강한 자간 — 한글 라벨에도 동일 적용) → Body/Button(Noto Sans KR Bold/Regular) → Caption(Noto Sans KR Regular, 뮤트 컬러). **세 개 폰트 패밀리(Baloo 2 / Inter / Noto Sans KR)를 역할별로 엄격히 분리해서 쓰는 것이 이 컨셉의 특징**이다 — 아무 텍스트에나 섞어 쓰지 않는다. design-systems가 텍스트 스타일 토큰을 만들 때 이 5단 분리를 그대로 유지한다.

## 4. 보더 & Radius 스케일

### 4-1. 보더 두께

- **3px** — 구조적 대분할선: 사이드바(main) 우측 보더, 메인 헤더 하단 보더
- **2px** — 기본 컴포넌트 보더: 버튼, 입력창, 모달/카드 외곽, count pill, 로고 원, 아바타 원, CornerInput 코너 브래킷
- **1px** — 경량 보더: CatBadge, row 액션 아웃라인 버튼(수정/삭제), 카테고리 관리 소형 버튼 보더(`#1c1f21`), 요약 박스 보더(`#ede6d8`, main-삭제), 경고 박스 보더(코랄, main-삭제), 테이블 헤더 하단(2px, ink) — 단 테이블 행 사이 구분선만 예외로 연한 베이지 `#ede6d8` 1px 헤어라인

### 4-2. Radius 스케일

- **999px(full round)** — 로고 원, 아바타 원, count pill, CatBadge 닷, TypeSelector 닷, 보조 버튼 아이콘 배지(20px)
- **10px** — 사이드바 nav 버튼, NeoInput/NeoBtn/NeoSelect(검색·필터·추가), 모달 저장/취소 버튼, 인증 폼 주요 버튼
- **8px** — 헤더 로그아웃 버튼, CatBadge, "총 N건" pill, row 액션 아웃라인 버튼(수정/삭제), TypeSelector 칩
- **5px** — 카테고리 관리 소형 액션 버튼(연필/휴지통)
- **6px** — 삭제 모달(main-삭제) 경고 박스
- **0px(각짐)** — 인증 폼(Join/login)·편집 모달(main-수정)의 CornerInput 입력창 자체(코너 브래킷 장식과 짝을 이루려 의도적으로 각짐 처리, main 화면 NeoInput은 반대로 radius 10 브래킷 없음 — 컨텍스트별로 다르게 처리한다), 삭제 모달 요약 박스, 알림 배너(성공/에러 둘 다 각진 사각형, 카드 시스템과 다르게 라운드 없음)

이 8단계(보더 3종 + radius 6종) 밖의 임의 값을 새로 추가하지 않는다.

## 5. 그림자 시스템 — 이전 문서의 "그림자 없음" 기록은 폐기, 하드/소프트 2종을 엄격히 구분

이전 파일럿 라운드(b-2-contacts-layout.md)에서는 "그림자를 전혀 쓰지 않는다"고 기록됐으나, **사용자의 최종 확정 디자인은 그림자를 적극적으로 쓴다.** 단, 용도가 다른 두 종류를 절대 섞어 쓰지 않는다 — **design-systems가 이 둘을 별개의 effect 토큰(예: `shadow/hard-sticker-{1,2,6}` vs `shadow/soft-float`)으로 분리 등록해야 한다는 점을 그대로 전달한다.**

### 5-1. 하드 오프셋 "스티커" 그림자 (neo-brutalist)

블러 없이 순수 오프셋, 색상은 항상 ink `#1a1a1a`. `drop-shadow(Npx Npx 0px #1a1a1a)`.

| 오프셋 | 적용 대상 |
|---|---|
| 1px | count pill(비활성) |
| 2px | 주요 CTA 버튼(검색/전체/추가/로그인/가입하기/저장하기/삭제하기), 활성 사이드바 nav, 모달 닫기(X) 버튼 |
| 6px | 모달 카드(main-수정, main-삭제), 인증 카드(Join, login) 전체 |

**적용 안 됨**(주요/보조 액션을 그림자 유무로 구분하는 일관 규칙): 보조/중립 버튼(취소, 로그아웃 NeoBtn, "로그인으로 돌아가기"), 비활성 nav, CatBadge, row 아웃라인 액션 버튼.

### 5-2. 소프트 블러 그림자 (토큰 기반)

알림 배너(성공 토스트 `247:5558`, 에러 배너 `247:5303`) **전용**. `0px 2px 8px rgba(28,31,33,0.08)`. 하드 스티커 그림자와 달리 오프셋 없이 부드럽게 퍼져, "일시적으로 떠 있다 사라지는" 알림의 성격을 카드/버튼과 다르게 표현한다.

## 7. 로고/워드마크 — 정확 재현 스펙

**심볼**: 지름 28px 원, 채움 `#ff5a76`(코랄), 2px ink(`#1a1a1a`) 스트로크, 내부에 흰색 `PxStar`(픽셀 별) 아이콘 12px 중앙 배치(패딩 2px). **빈 원이 아니다 — 항상 픽셀 별을 품는다.** 이 심볼 내부 별은 로고가 등장하는 모든 화면(main 사이드바 상단, Join/login 카드 상단, 회원가입 아이콘 배지 등)에서 빠짐없이 재현한다.

**워드마크**: "yourbook." — 전부 소문자 + 마침표(대문자 없음). 서체 **Baloo 2 ExtraBold**(Bold가 아니라 ExtraBold — 이전 b-2 문서의 "Bold" 기록과 다르니 혼동 주의, 이번 확정 디자인은 ExtraBold), 크기 22px, 자간 -0.55px(약 -2.5%, 살짝 좁힘).

**배경별 색 반전 규칙(중요)**: 워드마크 색상은 고정값이 아니라 **배경에 따라 반전된다**:
- 틸(`#17a398`) 배경 위(main 사이드바, Join/login 카드 밖 풀블리드 배경) → **흰색** ⚠ 2026-07-17 각주 — 이 배경은 실제로 스카이블루(`#1395e6`)로 전환됐다(2-1절 참고). 반전 규칙(브랜드색 배경=흰색, 흰 배경=ink) 자체는 그대로 유효하다.
- 흰 배경 위(카드 내부, 헤더) → **ink `#1a1a1a`**

고정된 딥틸 색(`#0F7A6E`) 하나만 쓴다는 이전 기록(b-2 문서 9-2절)은 이 확정 디자인에는 적용되지 않는다 — 화면마다 로고가 얹히는 배경을 먼저 확인하고 그 배경에 맞춰 반전시킨다.

**배치**: 심볼과 워드마크는 가로 배치, gap 10px, 아이템 정렬은 수직 중심 정렬(`items-center`).

**축소 시**: 심볼만 단독으로 쓸 경우(파비콘 등) 최소 크기에서도 2px 스트로크 비율은 유지하되, 내부 픽셀 별은 남긴다(관찰된 8개 프레임에서는 항상 심볼+별 세트로만 등장 — 스트로크를 제거한 별도 축소형은 이번 소스에 없다, 필요 시 design-systems가 파비콘 전용 변형을 정의).

## 8. 장식 모티프 (Decorative Motifs) — 화면별 적용 위치

- **픽셀 컨페티 배경(BgPixels)** — Join(`241:1552`)/login(`247:6666`)처럼 브랜드색(틸) 풀블리드 배경에서는 `PixelDiamond`/`PixelStar`/`PixelCross` 8bit 픽셀 아이콘 약 14개를 화면 전역에 8~20px 크기, 25~40% 불투명도로 무작위 산포시킨다. 여기에 점선 원(Ellipse, 대시 아웃라인) 3개를 카드 주변에 추가로 배치한다. 카드 안쪽 폼 하단에도 8px 크기 Diamond/Star 5개를 25% 불투명도로 한 줄 배치("컨페티 푸터").
- **픽셀 클러스터(MemphisAccents)** — main(`214:349`) 사이드바처럼 좁은 브랜드색 패널에서는 같은 픽셀 요소(Diamond/Star/Vector/Ellipse)를 화면 전역이 아니라 패널 하단 한 구석에 작게 클러스터로 모아 10% 컨테이너 불투명도(+개별 요소 35~67%)로 은은하게 배치한다. **풀블리드 배경(Join/login) = 전역 산포, 좁은 패널(main 사이드바) = 구석 클러스터**로 크기에 따라 밀도 규칙이 다르다.
- **코너 브래킷 인풋(CornerInput)** — Join/login과 main-수정(편집 모달)의 입력창은 각진(radius 0) 흰 박스 바깥 네 모서리에 8×8px, 2px ink 보더의 "ㄱ자" 브래킷을 얹어 "타겟팅/청사진" 느낌을 낸다. 이 장치는 인증/모달 필드에만 쓰고, main 화면의 검색·필터 NeoInput(10px 라운드, 브래킷 없음)에는 쓰지 않는다.
- **카드 상/하단 액센트 스트립** — main-수정/main-삭제 모달과 Join/login 인증 카드는 흰 카드 상단에 8px 두께 앰버(`#ffcb47`) 스트립을 얹는다(border-b 2px ink). Join/login 인증 카드는 여기에 더해 카드 **하단**에도 8px 두께 틸(`#17a398`) 스트립을 하나 더 얹어 위/아래를 다른 브랜드색으로 "샌드위치"한다 — 모달(main-수정/main-삭제)에는 이 하단 스트립이 **없다**(상단 앰버만). ⚠ 2026-07-16 재실측 정정 — 실제로는 스카이블루(#1395e6, color/sky/500)다. login(`501:5184`)/Join(`501:4936`) 둘 다 확인. 원문 "틸" 서술은 오기이므로 정정하되 삭제하지 않고 각주로 남긴다.
- **Count Pill / CatBadge** — main 화면 사이드바 nav의 원형 알약형 카운트 배지("총 N건")와, 테이블 행마다 붙는 연한 배경+보더+닷의 CatBadge(2-2절)가 리스트 전반에서 반복되는 핵심 장식 겸 정보 전달 장치다.
- **로고 심볼 내부 픽셀 스타** — 로고가 등장하는 모든 화면 공통(7절 참고).

## 9. 아이콘 계층

관찰된 아이콘은 명확히 두 계층으로 나뉜다:

1. **마이크로 픽셀 아이콘(8~15px, `Pixel*`/`Px*` 네이밍)** — 8비트/픽셀아트 렌더링, 단색(주로 ink 또는 흰색, 배경에 따라 반전), 안티앨리어싱 없는 각진 실루엣.
   - **기능 마이크로 아이콘**: `PxSearch`(검색), `PxPlus`(추가), `PxLogout`(로그아웃), `PixelPencil`(수정), `PixelTrash`(삭제), `PixelEye`(비밀번호 표시/숨김) — 버튼/입력창 안에 인라인으로 들어가는 9~15px 크기.
   - **순수 장식 컨페티**: `PixelDiamond`, `PixelStar`, `PixelCross`, `Ellipse`(점선 원) — 8절 참고.
2. **표준 플랫 아이콘(24px, `Icon/*` 네이밍, 컴포넌트화됨)** — 픽셀아트가 아닌 매끈한 플랫 실루엣. `Icon/User`(아바타 내부), `Icon/Alert`(성공 토스트/에러 배너 공용 — 배너 종류는 아이콘이 아니라 배경색으로만 구분한다). Graphic Assets(`90:2`) 참조.

**빈 상태(empty state) 그래픽**: `main-검색없음`(`242:4280`, 구 ID — 신규 확정 세트 기준 `501:4218`)에서 확인 — 별도의 큰 일러스트 대신 위 1번 계층의 픽셀 아이콘을 확대한 형태(깨진/기울어진 돋보기 실루엣, 틸+코랄 2색)를 재사용한다. 신규 마스코트를 만들지 않고 로고/기능 아이콘과 같은 픽셀아트 언어를 확장하는 방향이 이 컨셉과 맞는다. ⚠ 2026-07-16 각주 — 신규 확정 세트(`501:4218`) 재실측 결과 몸통은 스카이블루(#1395e6, color/sky/500)로 정정, 크랙은 코랄 그대로 유지("스카이블루+코랄 2색"). `Pixel/NoResult`(`255:149`, Icons 페이지) 마스터도 이 실측값에 맞춰 리바인딩 완료 — design-system.md 신규 절(B작업) 참고.

## 11. 알림/토스트 오버레이 — 의도된 플로팅 UI (버그 아님, 사용자 확인 완료)

login-알림창(`247:5303`)과 main-알림창(`247:5558`)은 **각각 login/main 화면 위에 최상위 z-index로 플로팅하는 의도된 오버레이**다 — 레이아웃 버그가 아니라 사용자가 직접 확정한 디자인 의도다. 후속 작업자(ui-designer/interaction-designer)는 두 배너 모두 **하단 콘텐츠를 밀어내지 않는 오버레이 방식**으로 구현한다. 자동 소멸 타이밍/애니메이션은 interaction-designer·motion-designer의 몫이며 이 문서는 정적 배치·스타일만 규정한다.

## 12. 텍스트 대비 규칙

브랜드 3색(틸/코랄/앰버)은 전부 mid-tone이라 흰 텍스트와 페어링하면 대비가 약할 수 있다 — 실제 확정 디자인에서도 앰버/코랄 배경 위 텍스트는 항상 ink(`#1a1a1a`)를 쓰고, 흰 텍스트는 틸(가장 어두운 브랜드색) 배경 위에서만 쓴다(사이드바, 로그아웃 버튼, 워드마크의 틸 배경 버전 등). 브랜드색을 배경으로 쓸 때 그 위 텍스트 색은 배경 밝기를 보고 ink/흰색 중 선택하는 규칙을 유지한다. ⚠ 2026-07-17 각주 — "틸 배경"은 실제로 스카이블루로 전환된 곳이 대부분이다(2-1절 참고). 규칙(브랜드색 배경=흰 텍스트, 흰 배경=ink 텍스트) 자체는 색이 바뀌어도 동일하게 유지된다.

## 13. 워드마크 네이밍 미확정 — 중요

"yourbook."은 서체·톤 표현용 placeholder이며 최종 서비스명이 아니다. 실제 서비스명 확정 시 design-pl 경유 별도 논의가 필요하다 — 어떤 워커도 임의로 확정하지 않는다.

## 14. design-systems / ui-designer를 위한 안내

- **토큰 추출 우선순위**: 8~10절의 화면별 실제 값을 근거로 Primitive→Semantic→Component 토큰을 추출한다. 파일럿(b-2 등)과 무관하게 이 8개 프레임에 실제로 쓰인 값만 추출 대상이다 — 없는 값을 임의로 보간하지 않는다.
- **하드/소프트 그림자를 반드시 별개 effect 토큰으로 분리**한다(5절). 하나의 "shadow" 토큰으로 합치면 알림 배너의 "떠 있다 사라지는" 성격과 카드/버튼의 "스티커" 성격이 구분되지 않는다.
- **TypeSelector vs CatBadge 색 불일치(2-3절)는 이 문서가 임의로 정리하지 않았다** — design-systems가 토큰화 단계에서 통일 여부를 판단하거나 design-pl 경유 사용자 확인을 받는다.
- **로고는 배경에 따라 반전(7절)** — 고정 색 하나로 컴포넌트를 만들면 안 되고, 브랜드색 배경/흰 배경 두 variant를 모두 등록한다(브랜드색은 현재 스카이블루, 2-1절 참고).
- **알림 오버레이 2개(11절)는 별도 컴포넌트로 등록하되 "플로팅 오버레이"라는 배치 성격을 컴포넌트 설명에 명시**해 이후 다른 화면에 확장할 때도 같은 배치 방식(하단 콘텐츠 안 밀어냄)을 유지하도록 한다.
- **⚠ 2026-07-17 신규 — Primary/Secondary/Accent 역할 재평가 반영**: 2-1절 각주·신규 행 참고. design-systems가 시맨틱 토큰(`color/bg-brand-primary` 등)을 명명할 때 Primary는 스카이블루, Secondary는 코랄, Accent는 앰버를 가리키도록 하고, 틸은 "카테고리 식별색" 스코프(예: `color/category/company-border`)로만 좁게 바인딩한다. 프리미티브 토큰 이름(`color/teal/500` 등) 자체는 바뀌지 않는다.
