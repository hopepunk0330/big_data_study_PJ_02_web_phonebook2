# 보고서 작성 형식 가이드

`docs/planning/`의 PDF 문서 세트(00~04)에서 공통으로 반복되는 **구조 패턴**을 정리한 것입니다. `docs/design/report-style.css`가 "시각적 스타일"이라면, 이 문서는 "글의 구조" 템플릿입니다. doc-writer가 시각화 보고서(HTML→PDF)를 작성할 때, 원본 PDF의 디자인을 직접 참조하는 대신 이 가이드의 구조와 `report-style.css`의 클래스를 사용합니다.

## 1. 문서 최상단 구조

모든 문서가 동일한 순서로 시작합니다.

1. `<h1>` 제목 — "{서비스명} — {문서 유형} ({과제 구분})" 형식
2. 메타데이터 표 (`table.meta`) — "항목 | 내용" 2열, 문서명/문서 유형/과제 구분/버전/상태 등을 담음
3. "이 문서의 역할(안내)" 콜아웃 (`.callout-info`) — 이 문서가 무엇을 다루고 무엇을 다루지 않는지, 관련 문서와의 관계를 1~2문단으로 설명

## 2. 섹션 번호 체계

- `<h2>`: "1. 제목", "2. 제목" — 최상위 섹션
- `<h3>`: "1-1. 제목", "3-2. 제목" — 부모 번호를 그대로 물려받는 하위 섹션
- 번호 없는 `<h3>`는 쓰지 않습니다 (추적/인용이 어려워짐).

## 3. 반복되는 콜아웃 박스 패턴

원본 PDF에는 색상별로 더 세분화된 박스가 있었지만, `report-style.css`는 두 종류(`.callout-info` 파랑, `.callout` 호박색)로 단순화했습니다. 아래처럼 매핑해서 씁니다.

| 패턴 | 용도 | 적용 클래스 | 문구 예시 |
|---|---|---|---|
| 문서 안내 | 이 문서/섹션이 무엇을 다루는지 오리엔테이션 | `.callout-info` | "이 문서를 읽는 분께(안내)", "PRD란?(안내)" |
| 설계 근거 | 왜 이 방식을 택했는지 설명 | `.callout` | "왜 이렇게 하나요? (설계 근거)" |
| 비유로 이해하기 | 낯선 개념을 익숙한 상황에 빗대어 설명 | `.callout` | "비유로 이해하기 — '헬스장 출입 카드와 프런트 장부'" |
| 실무 연결 | 배우는 내용이 실무에서 어떻게 쓰이는지 | `.callout` | "실무 연결" |
| 이전 버전과의 대응 | 이전 산출물과 무엇이 달라졌는지 | `.callout` | "1차 과제와의 대응", "1차 과제와 비교하면" |
| 문서화 근거 | 이 문서를 이렇게 구성한 이유 자체를 설명 | `.callout` | "~하는 이유(실무 관점):" |

콜아웃은 항상 **굵은 제목 한 줄 + 설명 1~3문장**으로 짧게 씁니다. 긴 서술은 본문으로 뺍니다.

## 4. 표 사용 패턴

세 가지 표 유형이 반복됩니다. 전부 `report-style.css`의 기본 `table` 스타일을 그대로 씁니다.

1. **ID 목록 표** — 요구사항/기능/화면마다 `FR-01`, `NFR-01`, `UC-01`, `AC-01`, `FN-001`처럼 접두어+번호 ID를 붙이고, "ID | 이름 | 설명 | 관련 항목" 형태로 나열합니다.
2. **상세 스펙 표** — 항목 하나(API 하나, 화면 하나, 함수 하나)를 "항목 | 내용" 2열로 세로 나열합니다 (요청/처리/성공/실패 등).
3. **비교 표** — "이전 방식 | 이번 방식 | (필요시) 실무에서 부르는 이름"처럼 좌우 대응을 보여줍니다.

## 5. 강조 문장 패턴

표나 코드 예시 바로 뒤에, 굵은 글씨로 핵심을 한 문장으로 요약하는 문장이 옵니다. 예: "**핵심 포인트는 이것입니다.**", "**주목할 점**: ...". 이 문장은 별도 박스가 아니라 본문 단락 안에 `<strong>`으로 넣습니다.

## 6. 다이어그램 표현 방법 (Mermaid 그대로 사용)

`pdf-maker/make-pdf.js`가 PDF 변환 시 mermaid.js를 로드해서 실제 그림으로 렌더링한 뒤 찍으므로, 원본 MD 문서의 Mermaid 다이어그램은 표나 ASCII로 바꾸지 않고 **그대로 옮깁니다.** 다만 코드 블록(```` ```mermaid ````)이 아니라 mermaid.js가 인식하는 형식으로 감싸야 합니다.

```html
<pre class="mermaid">
flowchart TB
    A[시작] --> B[끝]
</pre>
```

- `class="mermaid"`가 붙은 `<pre>` 태그 안에 다이어그램 원본 코드(flowchart/erDiagram/sequenceDiagram 등)를 그대로 넣습니다.
- 다이어그램 종류나 문법을 바꾸지 않습니다 — 원본 MD의 Mermaid 코드를 그대로 복사해 넣으면 됩니다.
- 트리 구조(파일 구조 등)처럼 원래 Mermaid가 아니라 ASCII 코드 블록(````text````)으로 되어 있던 것은 그대로 `<pre><code>`로 유지합니다(이건 Mermaid가 아니므로 변환 대상이 아닙니다).

## 7. 문서 마무리 패턴

문서 끝에는 아래 중 하나(또는 복수)가 옵니다.

- **점검 체크리스트** — `- [ ] 항목` 형태
- **추적표(Traceability)** — 이 문서의 ID가 다른 문서 어디에 대응하는지 표로 정리
- **후속 문서 안내** — 문서 세트 전체 목록과 읽는 순서 표

## 8. 꼬리말

각 페이지 하단에 "{문서명} · {페이지}/{전체페이지}" 형식의 작은 회색 텍스트가 있었습니다. 페이지 번호 자동 계산은 현재 `pdf-maker/make-pdf.js`가 지원하지 않으므로, `.doc-footer` 클래스로 문서 맨 끝에 한 번만 "{문서명} · {버전}"을 적는 것으로 대신합니다. (페이지별 자동 번호가 필요해지면 `make-pdf.js`에 `displayHeaderFooter`/`footerTemplate` 옵션을 추가하는 별도 작업이 필요합니다 — 지금은 범위 밖입니다.)

## 9. doc-writer 적용 체크리스트

보고서 HTML을 쓸 때 아래를 순서대로 확인합니다.

- [ ] 이 문서는 사람이 읽는 최종 산출물임을 인지하고, 원문 내용을 임의로 요약·생략하지 않았는가
- [ ] `<h1>` + `table.meta`로 시작했는가
- [ ] "문서 역할" `.callout-info`를 넣었는가
- [ ] 섹션 번호가 `1.` → `1-1.` 체계를 따르는가
- [ ] 설계 근거·비유·실무 연결 중 필요한 것만 `.callout`로 넣었는가 (과하게 넣지 않는다 — Simplicity First)
- [ ] 표가 ID 목록/상세 스펙/비교 표 중 맞는 유형을 쓰는가
- [ ] Mermaid 다이어그램이 `<pre class="mermaid">...</pre>` 형태로 원본 그대로 들어있는가 (표/ASCII로 바꾸지 않았는가)
- [ ] 문서 끝에 체크리스트/추적표/안내 중 해당하는 마무리가 있는가
- [ ] `report-style.css` 전체를 `<style>`에 인라인했는가
