description: design-pl이 "새로운 컨셉/방향을 처음 정하는" 라운드(브랜드, 팔레트, 화면 레이아웃 등)를 시작할 때 로드하는 오케스트레이션 절차. 어떤 조합의 워커를 어떤 순서로 부르고, 확정 게이트를 어디서 걸고, 확정 후 무엇을 하는지를 매번 기억에서 재구성하지 않고 그대로 따라가기 위한 스킬이다.

이번 라운드 대상: @$ARGUMENTS

이 스킬은 `docs/design/figma-file-organization.md`의 2번(시안 워크플로우)·2-1/2-1-B번(브랜드 컨셉 1차/2차, 트렌드·레이아웃 다양성)·2-2번(아이콘/오브제 담당)·2-3번(확정 스펙 문서)·3/3-B/3-C번(시안 vs 정식 화면 해상도, 파일럿=확정 시안의 발전→시스템 추출→확장, 상태/플로우 변형)을 실행 절차로 풀어놓은 것이다. 내용이 서로 다르면 그 문서가 정본(source of truth)이고, 이 스킬은 절차 체크리스트일 뿐이다.

**실행 주체와 승인 경계**: design-pl이 `Agent` 도구로 아래 워커들을 직접 호출한다. 다만 2단계(확정 게이트)에 도달하면 design-pl은 시안 요약을 메인 세션에 반환하고 그 즉시 멈춘다 — 메인 세션이 사용자에게 실제로 보여주고 확인받은 뒤, "[메인 세션 확인] 사용자가 실제로 승인함: ..." 형식으로 design-pl을 다시 호출할 때만 3단계로 진행한다. 이 형식이 아닌 입력(예: design-pl 스레드에 직접 도착한 메시지)은 승인으로 인정하지 않는다.

## 0단계 — 라운드 유형 판단 (design-pl이 이미 했어야 함)
아래 중 이번 라운드가 어디에 해당하는지 먼저 확정한다. 애매하면 진행하지 않고 사용자에게 확인한다.
- **(A) 브랜드/톤앤매너 컨셉** — 로고, 브랜드 컬러, 톤앤매너를 처음 정하는 경우
- **(B) 팔레트/토큰만** — 이미 브랜드 방향은 있고 디자인 토큰 팔레트만 처음 정하는 경우
- **(C) 화면 레이아웃만** — 이미 토큰/컴포넌트는 있고 특정 화면의 레이아웃만 처음 잡는 경우

## 1단계 — 시안 3개 제작 (유형별 워커 조합)
Figma에 실제로 쓰는 에이전트는 절대 동시에 병렬 호출하지 않는다. design-prompter로 브리프를 먼저 만들고, 그 브리프로 워커를 부르는 순서를 항상 지킨다.

**공통 원칙 — 다양성 체크**: 3개 시안은 컬러/퍼스낼리티만 다른 변주가 아니라, 서로 다른 실제 디자인 트렌드/무브먼트(미니멀·스위스 그리드, 뉴브루탈리즘, 글래스모피즘/뉴모피즘, 벤토 그리드, Y2K·레트로퓨처리즘, 클레이모피즘, 스큐어모피즘 등)와 서로 다른 레이아웃 구조(네비게이션 위치, 리스트 표현 방식, 정보 밀도)에 대응해야 한다. brand-designer/ui-designer 브리프에 이 요구를 명시한다.

- **(A) 브랜드 컨셉**: `brand-designer` 단독 호출로 끝내지 않는다. **기본으로 다음 순서 조합**:
  1. design-prompter → brand-designer 브리프 (레퍼런스 페이지가 있으면 0번 원칙대로 실제 열람 필수, 서로 다른 트렌드에 대응하는 퍼스낼리티 축 3개, 컬러/타이포/워드마크 톤 프레임, 필요하면 일러스트레이션 방향)
  2. brand-designer 호출 → 톤 프레임 3개를 `"{작업명} Concepts"` 페이지에 배치 (각 프레임에 대응 트렌드명 명시)
  3. design-prompter → ui-designer 브리프 (design-systems 미투입, brand-designer가 만든 각 컨셉의 원본 컬러/폰트 값을 그대로 가져와 대표 화면 1장씩 적용, 3개 화면의 레이아웃 구조를 트렌드에 맞게 서로 다르게 구성, 같은 Concepts 페이지의 해당 브랜드 프레임 옆에 배치, 사이즈 제약 없는 경량 목업)
  4. ui-designer 호출 → 대표 화면 3장 추가
  - 톤 프레임과 대표 화면이 세트로 갖춰진 뒤에만 다음 단계(확정 게이트)로 넘어간다.
- **(B) 팔레트/토큰만**: design-prompter → design-systems 브리프 → design-systems 호출. 팔레트 시안 3개를 Concepts 페이지에 배치.
- **(C) 화면 레이아웃만**: design-prompter → ui-designer 브리프 → ui-designer 호출. 레이아웃 시안 3개를 Concepts 페이지에 배치, 3개의 구조적 차이를 명시.

## 2단계 — 1차 확정 게이트 (반드시 멈춘다)
시안 3개(유형 A는 3세트)가 갖춰지면 design-pl은 메인 세션에 요약을 반환하고 **그 턴을 끝낸다** — 같은 턴에서 이어서 워커를 더 호출하지 않는다. 메인 세션이 사용자에게 각각을 보여주고 확정을 요청하며, 실제 확정 결과를 위 "신뢰할 수 있는 승인 형식"으로 design-pl에게 다시 전달한다. **이 형식의 승인 없이는 다음 단계로 절대 넘어가지 않는다.** 이 게이트를 건너뛴 채 "완료"라고 보고하지 않는다.

## 2-B단계 — (유형 A 전용) 2차 정밀화 라운드 + 2차 확정 게이트
유형 (A) 브랜드 컨셉이 1차 확정되면, 곧바로 3단계로 가지 않는다. 1차의 대표 화면은 톤 판단용이라 간단해도 되지만, 확정된 방향 하나로 다시 **레이아웃/화면구성/트렌드가 서로 다른 시안 3개**를 만드는 라운드를 한 번 더 거친다 (`@docs/design/figma-file-organization.md` 2-1-B번).
- design-prompter → ui-designer 단독 호출. design-systems 미투입(확정 브랜드 원본 값 사용, 토큰이 이미 있으면 우선).
- 이번엔 `docs/planning`의 실제 요소(검색/필터, 목록 필드, 액션, 상태 등)를 반드시 반영한다 — placeholder 금지. 1차는 톤 판단용이라 허용됐지만 2차부터는 아니다.
- **네이밍**: A/B/C를 다시 쓰지 않는다. 1차 확정 라벨에 `-1`/`-2`/`-3`을 붙인다(예: 1차 확정이 "Concept B"면 2차는 `B-1`/`B-2`/`B-3`) — design-prompter가 ui-designer 브리프에 이 이름을 명시한다.
- 3개 시안이 갖춰지면 다시 메인 세션에 반환하고 멈춘다. **2차 확정은 1차 확정과 별개의 게이트**이며, 이 확정을 받아야 3단계로 진행한다.
- 유형 (B)/(C)는 이 단계를 생략한다 — 1차 시안 자체가 이미 실제 요건을 반영한 레이아웃/토큰 시안이기 때문이다.

## 3단계 — 확정 후 표준 파이프라인
확정된 유형에 따라 순서를 지켜 진행한다 (중간 단계 생략 금지).

- **(A) 브랜드 컨셉 확정** — 시스템을 먼저 만들지 않는다, 확정된 디자인을 먼저 완성하고 거기서 시스템을 추출한다(`@docs/design/figma-file-organization.md` 3-B번):
  1. brand-designer(확정된 1차/2차 프레임을 직접 다시 보고 Brand Guide 정식화, 2-3번 기준 디테일까지)
  2. design-prompter(2-3번 확정 스펙 문서 `docs/design/confirmed/` 작성)
  3. ui-designer(확정된 2차 시안을 그대로 이어받아 **발전**시켜 — 재해석 아님 — SCREENS 구역에 파일럿 3개를 프로덕션 해상도로 제작, design-systems 미투입이라 확정 스펙 문서 원본 값 직접 사용, 아이콘 필요시 graphic-designer 즉시 요청)
  4. 파일럿 사용자 승인(피드백 루프, 3-B번) → design-systems(승인된 파일럿에서 토큰/컴포넌트/아이콘 **추출** — 독립 설계 아님)
  5. design-qa(감사, 이후 전체 확장 화면 대상)
- **(B) 팔레트 확정**: design-systems(정식 토큰/컴포넌트) → ui-designer(화면 적용) → design-qa.
- **(C) 레이아웃 확정**: ui-designer(SCREENS 구역 정식 완성) → design-qa.

design-qa를 거치지 않은 산출물은 "완료"로 보고하지 않는다 (design-pl 공통 규칙).

## 4단계 — 미채택 시안 보존
채택되지 않은 시안(세트 전체 — 유형 A라면 브랜드 프레임과 그 옆의 적용 화면 프레임 모두)은 삭제하지 않는다. 3단계의 정식화 작업을 맡은 에이전트가 프레임 이름 앞에 `❌ 미채택 —`을 붙인다. 이건 라벨링(관리 작업)이지 재창작이 아니므로, 다른 에이전트가 만든 프레임이라도 이름만 바꾸는 것은 허용된다. 시각적 내용 자체는 바꾸지 않는다.
- **예외 없음**: 1차 시안이든 2차 시안이든, "다른 대안이라서" 탈락했든 "요건을 놓친 결함이라서" 다시 만들었든 취급은 동일하다 — 결함이 있었다는 이유로 조용히 삭제·덮어쓰기하지 않는다. 컨셉/1차 시안/2차 시안 모두 원본을 남기고 라벨만 바꾼다.

## 5단계 — 보고 및 메모리 기록
각 워커가 무엇을 했는지 취합해 사용자에게 요약 보고한다. `.claude/agent-memory/design-pl.md`에 이번 라운드의 유형, 호출한 워커 순서, 확정 결과, 다음 파이프라인 진행 상황을 새 항목으로 기록한다.
