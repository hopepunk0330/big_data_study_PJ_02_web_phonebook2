### 2026-07-15 — "로그인 상태 유지" 체크박스 흐름 검토 + 8개 확정 프레임 재스캔 (읽기 전용, 수정 없음)
- 체크박스(247:6822~6825, login 247:6666): 01(v1.2)·02(v1.5)·03(v1.2)·04(v1.3) 전체를 다시 확인 — "로그인 상태 유지"/"remember" 관련 기능 요구사항이 4개 문서 어디에도 없음. 01 §1 `sessions` 테이블에 만료(expires_at) 컬럼 자체가 없고, FR-02 로그인 처리(①~⑤)에도 지속시간 분기가 없음. 게다가 G4/PR-11("브라우저를 닫았다 다시 열어도, 서버를 재시작해도 데이터가 그대로")이 이미 로그인 상태를 무조건 세션 삭제 전까지 유지한다고 규정 — 체크박스가 켜져 있든 아니든 결과가 이미 같다는 뜻. "상태를 저장하겠다"고 약속하는 UI인데 그 상태가 실제로 다르게 동작한다는 근거가 문서에 없어 시스템 상태 가시성 위반 소지 — design-pl에 보고, 01 문서 보완 여부는 사용자 판단 필요(임의로 삭제/추가 안 함).
- "비밀번호 찾기→재설정" 건: 8개 프레임 재확인 결과 이미 전부 "비밀번호 재설정"으로 통일되어 있고 02문서도 v1.5로 동기화 완료 — 재작업 불필요(중복 보고 안 함).
- 신규 발견 3건(design-pl 보고용, 미수정):
  1. main-검색없음(242:4280): "검색 결과가 없습니다" 상태인데 상단 배지가 여전히 "검색결과 총 6건"으로 표시됨(실제 결과 0건과 불일치) — main 프레임을 복제하며 헤더 갱신을 빠뜨린 것으로 보임.
  2. main(214:349) 등 목록 화면 전체: "전체" 카테고리를 선택한 기본 목록 상태에서도 헤더 라벨이 "검색결과"로 고정 표시됨 — 02문서는 이 라벨을 "총 N건"으로만 정의하고 "검색결과"라는 접두어를 쓰지 않음. 실제로 검색을 안 한 상태에서도 "검색결과"라고 보이면 사용자가 이미 필터링된 목록을 보고 있다고 오인할 수 있음(일관성과 표준/실제 세계와의 매칭 위반 소지).
  3. main-삭제(248:9867): 확정 디자인은 커스텀 "연락처 삭제" 모달(경고 아이콘+상세정보+삭제하기/취소 버튼)인데, 02문서 SCR-002 표·Mermaid는 여전히 "확인 대화상자(confirm)"로 브라우저 native `confirm()`을 가리키는 문구로 남아있음 — 앞서 이미 처리된 "연락처 수정=모달" 사례(01 항목 (a), v1.1)와 같은 성격의 문서-디자인 불일치. 같은 방식으로 문서 갱신할지 design-pl·사용자 판단 필요.
- 접근성: 체크박스 클릭 영역(247:6822, Button 래퍼 105×18px, 실제 체크박스 시각 요소는 14×14px)이 WCAG 권장 최소 타깃(24×24) 미달 — 시각 크기/색상은 design-systems 소관이라 다루지 않되, 클릭 가능 영역 확대는 후속 검토 권장. 체크박스와 라벨("로그인 상태 유지")이 같은 Button 그룹 안에 있어 라벨 클릭도 토글되는 구조로 보이는 점은 접근성상 긍정적(라벨-컨트롤 연결).
