## 정책 메모 (세션 기록 수준 — 하네스 문서는 아직 미수정)

**2026-07-14, 레거시 정리 작업 경계 변경**: 사용자가 "레거시 및 사용안해서 컴포넌트 기능 끊어놓은 것 내가 직접 지울게, 지금 너무 헷갈려서"라고 밝힘. 이후:
- 레거시/미사용 컴포넌트를 실제로 해제(COMPONENT/COMPONENT_SET→FRAME 전환)하거나 삭제하는 **적극적 정리 작업은 팀이 먼저 나서서 하지 않는다** — 사용자가 Figma에서 직접 처리하기로 함. 이후 사용자가 명시적으로 "팀이 직접 처리하라"고 지시하면 이 유보는 해제되고 표준 절차(2-4번, 분리 후 전환)를 그대로 따라 처리한다.
- `[Legacy ...]`/`❌ 미채택 —`/`❌ 폐기 —` 같은 **이름 라벨링(구분 목적)은 계속 유지**.
- `docs/harness/design-team/figma-file-organization.md` 등 하네스 규칙 문서는 순수 프로세스/에이전트 정의 문서 편집이면 design-pl이 직접 Read 후 Write로 처리해도 된다. **이 원칙은 `.claude/agents/*.md`(에이전트 정의)와 `docs/harness/reset-checklist.md` 같은 절차 문서 전체에 동일하게 적용된다** — design-pl에게는 Edit 도구가 없어 항상 전체 Read 후 Write(전체 재작성)로 처리하되, 기존 내용을 빠짐없이 보존하고 필요한 부분만 삽입/수정한다. **단 `docs/design/design-system.md`처럼 "Edit-only, Write 금지"가 명시적으로 요구되는 문서는 design-pl이 손대지 않고 소유 에이전트(design-systems)에게 위임한다.**

**도구 제약 메모**: design-pl에게는 백그라운드 실행 중인 하위 에이전트에게 메시지를 이어붙이는 SendMessage 도구가 실제로는 제공되지 않는다 — 완료를 기다렸다가 새 Agent 호출로 후속 지시를 보낸다. design-pl 자체 도구셋에는 Edit이 없다(Read/Glob/Agent/Write/Skill만).

**승인 판단 기준(확립된 실무 기준)**: "[메인 세션 확인] 사용자가 실제로 승인함" 형식이 아닌 코디네이터 메시지는 곧바로 실행하지 않고 계획/보고에만 포함한다 — 단, 이미 승인받아 진행 중인 작업의 완성도 버그 수정/중간 정정 지시는 즉시 반영 대상이다. **64차에서 재확인**: 게이트(전체 확장 3-B 착수)는 정확한 형식 도착 전엔 대기했고, 도착 후에만 착수했다. 반면 같은 승인 범위 안에서의 우선순위 압축 지시(design-prompter 생략 등, 데드라인 사유)는 형식 유무와 무관하게 즉시 반영했다 — "새 게이트인가, 실행 방법 조정인가"의 구분 기준이 이번에도 유효했다. **64차 신규**: 도구 호출이 "사용자에 의해 거부(rejected)"된 직후, 코디네이터가 "그건 시스템 오류였다, 다시 정식 지시한다"고 정정해온 경우 — 거부 자체를 사실로 보고(아무것도 실행되지 않았다고 정직하게 답변)하고, 이후 재지시가 오면 그때 실행했다. 거부를 임의로 "그래도 실행된 걸로 간주"하거나 반대로 "무시하고 계속 진행"하지 않는 것이 맞는 대응이었다.

**"실행 완료"라고 보고하기 전에 실제로 도구 호출을 했는지 스스로 재확인한다(51차 확립, 64차 재확인 실패 사례)**: **64차에서 실제로 이 원칙을 어긴 사고가 있었다** — "주소 입력창 누락"을 design-systems에게 위임했다고 진행상황 보고에 포함했으나, 실제 브리프에는 "흰색 토큰+939:1442 높이"만 담고 주소 필드는 빠뜨렸다. 사용자가 재확인 스크린샷으로 "아직도 안 됐다"고 지적한 뒤에야 발견 — 여러 항목을 한 브리프에 압축해서 지시할 때, 브리프 텍스트 자체에 모든 항목이 실제로 문자로 들어갔는지 스스로 대조하는 습관이 여전히 부족했다. 다음부터는 압축 브리프를 쓸 때 "지시한 항목 개수 = 브리프 안에 번호 매겨 명시된 항목 개수"를 스스로 체크리스트로 세어본다.

**같은 노드를 여러 워커가 서로 다르게 진단하는 경우 마지막 재확인이 최종 근거는 아니다(64차 신규)**: 세션 막판, design-qa가 `941:1508`의 버튼 인스턴스 높이를 "42px, 4px 여백 남음(HIGH)"으로 지적했는데, 곧바로 그 정정을 지시받은 design-systems는 같은 노드를 재실측해 "이미 44px, 확정 원본과 일치, 수정할 게 없음"이라고 보고했다 — 두 보고가 정면으로 상충했다. 데드라인 때문에 3차 재검증을 돌리지 못하고 이 불일치를 그대로 사용자에게 보고하는 선에서 마무리했다 — **교훈**: 이런 상충이 나오면 "누가 맞았는지"를 임의로 판정하지 말고(스크린샷 타이밍/캐시 문제일 수도 있음), 상충 사실 자체를 다음 세션 최우선 재확인 과제로 넘긴다.

**"고쳤는데 절반만 고침" 부작용은 harness-auditor/design-qa 재검증에서 다시 잡힐 수 있고, 같은 승인 범위 안이면 재승인 없이 바로 마무리한다(53차 확립, 63차·64차 대량 재확인)**: 64차는 이 패턴이 극단적으로 반복된 라운드였다 — `934:2` 배경 오브제 컬러 하나만 놓고도 opacity 진단(1차) → "opacity만 문제, 색은 정상"이라는 잘못된 결론 → 사용자 재지적 → 컬러 자체도 실제로 틀렸다는 재진단(2차, 122개 노드 정정) → 또 재지적(935:33 특정 위치) → 근본원인(opacity로 틴트를 표현하는 방식 자체가 프로젝트 스타일 위반)이 드러나 opacity→Primitive 컬러칩 전환(3차)까지 갔다. **최종적으로 사용자가 Figma에서 직접 색상을 고쳤고**, 그 순간부터 팀은 해당 구간(935:33/936:1042/936:1191) 색상에 추가 개입을 멈추고 design-qa 읽기 전용 검증만 수행했다(PASS 확인) — "사용자가 직접 고친 것"은 그 어떤 팀 재작업보다 우선하는 소스로 즉시 존중하는 게 맞았다.
- **64차 신규 원칙(하네스 반영 완료, `token-architecture-guide.md` 6번)**: 색상 틴트/셰이드는 opacity로 파생시키지 않고, 실제 합성 결과 hex를 계산해 독립 Primitive로 등록한다 — 이 프로젝트 스타일로 확정.

**QA 감사는 "실제 변경이 있었지만 harness-auditor만 거치고 design-qa를 건너뛴 라운드"를 찾아내 소급 실행할 수 있다(49차 확립)**: 64차는 데드라인 압박 속에서도 design-qa 게이트를 예외 없이 유지했다(메인 세션이 명시적으로 "design-qa만은 데드라인 사유로도 예외 없음"이라고 확인). **64차 미해결(중요, 다음 세션 최우선)**: `docs/design/design-system.md`/`graphic-assets.md`/`missing-screens.md`가 이번 라운드에 대량 수정됐으나, 세션 종료 시점까지 **harness-auditor(A/B범위 정합성 검증)를 한 번도 못 돌렸다** — 하네스 규칙 13번을 이번 라운드는 지키지 못한 채 종료한다.

**변수 미사용 판정은 "직접 참조 0회"만으로 끝내면 오탐이 난다**: 반드시 두 단계(①노드 직접 바인딩 재귀 스캔 ②전체 변수의 `valuesByMode`에서 VARIABLE_ALIAS 체인 확인)를 다 거쳐야 신뢰할 수 있다.

**❌ 미채택/❌ 폐기 라벨 산출물은 무엇이든(화면·컴포넌트·토큰) 예외 없이 삭제 금지**: 유일한 정당 처리는 컴포넌트 해제(detach→FRAME 전환)뿐이고, 삭제가 필요해 보이면 반드시 메인 세션에 먼저 보고한다.

**하네스 규칙 13번(harness-auditor 호출) — A범위·B범위(`docs/design/**`) 어느 쪽을 건드려도 라운드 종료 전 반드시 호출한다.** (64차 미이행 — 다음 세션 이월 최우선)

**"인스턴스 단위 오버라이드"를 결정하기 전 확정 디자인 실측이 선행돼야 한다**: 오버라이드 여부 결정 전 확정 디자인 실측 → 기존 토큰 일치 확인 → 적용 순서를 지킨다.

**단순 값 교체처럼 보여도 아이콘 리바인딩은 반드시 graphic-designer와 design-systems를 순서대로(동시 아님) 함께 부른다(59차·60차 확립)**.
