# design-pl 메모리

이 파일은 design-pl이 작업 시작 시 읽고, 작업 종료 시 기록을 남기는 메모리/로그입니다.

## 팀 로스터 (참고용)
- design-prompter, design-scanner, brand-designer, graphic-designer, design-systems, ux-designer, ui-designer, interaction-designer, motion-designer, content-designer, design-qa

## 정책 메모 (세션 기록 수준 — 하네스 문서는 아직 미수정)

**2026-07-14, 레거시 정리 작업 경계 변경**: 사용자가 "레거시 및 사용안해서 컴포넌트 기능 끊어놓은 것 내가 직접 지울게, 지금 너무 헷갈려서"라고 밝힘. 이후:
- 레거시/미사용 컴포넌트를 실제로 해제(COMPONENT/COMPONENT_SET→FRAME 전환)하거나 삭제하는 **적극적 정리 작업은 팀이 먼저 나서서 하지 않는다** — 사용자가 Figma에서 직접 처리하기로 함. 이후 사용자가 명시적으로 "팀이 직접 처리하라"고 지시하면 이 유보는 해제되고 표준 절차(2-4번, 분리 후 전환)를 그대로 따라 처리한다.
- `[Legacy ...]`/`❌ 미채택 —`/`❌ 폐기 —` 같은 **이름 라벨링(구분 목적)은 계속 유지**.
- `docs/harness/design-team/figma-file-organization.md` 등 하네스 규칙 문서는 순수 프로세스/에이전트 정의 문서 편집이면 design-pl이 직접 Read 후 Write로 처리해도 된다. **이 원칙은 `.claude/agents/*.md`(에이전트 정의)와 `docs/harness/reset-checklist.md` 같은 절차 문서 전체에 동일하게 적용된다** — design-pl에게는 Edit 도구가 없어 항상 전체 Read 후 Write(전체 재작성)로 처리하되, 기존 내용을 빠짐없이 보존하고 필요한 부분만 삽입/수정한다. **단 `docs/design/design-system.md`처럼 "Edit-only, Write 금지"가 명시적으로 요구되는 문서는 design-pl이 손대지 않고 소유 에이전트(design-systems)에게 위임한다.**

**도구 제약 메모**: design-pl에게는 백그라운드 실행 중인 하위 에이전트에게 메시지를 이어붙이는 SendMessage 도구가 실제로는 제공되지 않는다 — 완료를 기다렸다가 새 Agent 호출로 후속 지시를 보낸다. design-pl 자체 도구셋에는 Edit이 없다(Read/Glob/Agent/Write/Skill만). **65차에서 재확인**: 백그라운드(`run_in_background: true`)로 보낸 순수 텍스트 정정 지시가 완료 통지 없이 조용히 무반영으로 끝난 사례가 있었다(harness-auditor 재검증에서 "미해소"로 드러남) — 백그라운드 호출 뒤 반드시 후속 감사로 실제 반영 여부를 재확인하고, 반영 안 됐으면 포그라운드로 재지시한다.

**승인 판단 기준(확립된 실무 기준)**: "[메인 세션 확인] 사용자가 실제로 승인함" 형식이 아닌 코디네이터 메시지는 곧바로 실행하지 않고 계획/보고에만 포함한다 — 단, 이미 승인받아 진행 중인 작업의 완성도 버그 수정/중간 정정 지시는 즉시 반영 대상이다. **65차 신규**: 메인 세션이 "사용자가 제출 마감을 연장했다"며 이전에 보류했던 항목(harness-auditor 소급 검증, 941:1508 상충 해소, missing-screens.md 5/7번)을 정식 절차로 이어가라고 지시 — 새 게이트가 아니라 이미 승인된 라운드의 재개이므로 즉시 반영했다. 이후 "dev-pl 쪽 frontend-engineer 작업을 도우라"(design-scanner 투입), "design-qa로 실구현 vs Figma 교차검수"도 같은 방식(신뢰 형식, 새 게이트 아닌 협업 요청)으로 즉시 반영했다. **65차 마무리 사례**: design-qa가 지적한 HIGH1(Join 화면이 login과 다른 세부 디테일 없이 통합 구현됨)에 대해, 메인 세션이 "사용자가 이미 '지금처럼 통합 유지'를 명시적으로 결정했다"는 신뢰 형식 메시지를 보내와, 이건 버그가 아니라 **의도적 스코프 결정**임을 확인 — design-qa의 지적이 사실관계로는 맞아도, 그 차이를 어떻게 취급할지(고칠 대상 vs 확정된 결정)는 design-pl이나 design-qa가 임의 판단하지 않고 항상 사용자 결정을 기다린다는 원칙이 재확인됐다.

**"실행 완료"라고 보고하기 전에 실제로 도구 호출을 했는지 스스로 재확인한다(51차 확립, 64차 재확인 실패 사례)**: 64차의 "브리프에 항목 누락" 실수 이후, 65차에서는 압축 브리프 대신 항목별로 완료 여부를 스스로 재확인하는 습관을 유지했다(예: 백그라운드 문서 정정 지시가 실제로 반영됐는지 harness-auditor로 재확인 → 미반영 확인 → 포그라운드로 재지시 → 재확인).

**같은 노드를 여러 워커가 서로 다르게 진단하는 경우, 반드시 한 번 더 신중하게 재확인해서 결론을 확정한다(64차 신규, 65차 해소)**: `941:1508` 버튼 높이 상충(design-qa "42px HIGH" vs design-systems "44px 문제없음")을 65차에서 design-systems가 노드ID를 처음부터 다시 특정하고 확정 원본과 교차검증한 결과 **44px가 최종 확정**됐다 — design-qa의 42px 보고는 stale 캐시/노드 착오로 추정, 재현 안 됨. 상충이 나오면 "누가 맞았는지" 섣불리 판정하지 않되, 여유가 있으면 반드시 한 번 더 신중한 재확인 라운드를 거쳐 확정 결론을 내는 게 맞다(데드라인으로 확정 못 하면 이월, 시간이 있으면 반드시 확정).

**design-scanner 같은 순수 조사 에이전트의 조사 결과도 그대로 믿지 않고 이번 세션의 실제 작업 이력과 대조한다(65차 신규 확립)**: dev-pl 협업 요청으로 design-scanner를 투입해 Figma 전체 인벤토리를 조사시켰는데, "Figma Variable 0개"(실제 122개 존재, `search_design_system`이 퍼블리시 라이브러리만 조회하고 로컬 변수를 못 찾는 도구 한계였음), "Button/NeoBtn 등 12개 컴포넌트 마스터 nodeId 미기록"(실제로는 design-system.md 5절에 전부 기록돼 있었음), "Graphic Assets 페이지 삭제됨"(이건 맞았음 — 2026-07-14 사용자가 실제로 삭제) 등 상당수가 틀린 결론이었다. design-systems에게 별도로 사실 검증을 시킨 결과 4개 항목 중 3개가 오탐, 1개(`SummaryBox` 미문서화)만 진짜 갭이었다 — **"판단 없이 조사만" 하는 에이전트라도, 그 조사에 쓴 도구 자체의 한계(라이브러리 검색 vs 로컬 변수 조회처럼 경로가 다른 도구)로 결론이 왜곡될 수 있다는 걸 항상 의심하고, 특히 이번 세션에서 이미 직접 확인한 사실과 모순되면 그대로 다음 워커에게 넘기지 않고 먼저 검증한다.**

**실제 프론트엔드 구현(코드/스크린샷) vs Figma 확정 디자인의 교차검수는 design-qa의 정규 업무 범위다(65차 신규 확립, 같은 차수 내에서 자기 정정 사례로 재확인)**: dev-pl 쪽에서 프론트엔드 재추출·회귀테스트를 마친 뒤, design-qa에게 `docs/screenshot/`의 실제 캡처 파일을 Figma와 나란히 대조시켰다. dev-pl이 "이미 수정했다"고 주장한 항목(로고 누락, 이모지 아이콘)도 실제 스크린샷을 다시 열어보니 일부는 재발/미해소 상태였다 — "수정 완료" 보고를 그대로 믿지 않고 산출물(스크린샷)로 직접 재확인하는 원칙이 팀 경계(디자인팀 vs 개발팀)를 넘어서도 동일하게 적용된다. **단, 이 원칙은 감사자 자신의 오탐 가능성도 동일하게 적용받는다** — HIGH2(로고 마스코트 미렌더링) 지적을 dev-pl이 "4곳·11개 상태 순회로 재현 안 됨"이라고 반박했을 때, design-qa에게 정확한 파일명·좌표까지 짚어 재확인시켰더니 처음엔 "재현됨"으로 재확인했으나, dev-pl이 스크린샷을 전량 재생성한 뒤 다시 확인하자 **28px 소형 로고를 축소 렌더링에서 안티에일리어싱으로 오독한 자기 오탐**이었음을 design-qa 스스로 인정·정정했다. 감사자의 "재확인 후에도 유효하다"는 결론도 최종 확정이 아니며, 상대측이 구체적 반증(전량 재캡처, 원본 해상도 재확인)을 제시하면 한 번 더 검증하고 필요하면 스스로 뒤집는 게 맞다. **또한 "사실관계가 맞다"는 것과 "그게 고쳐야 할 문제다"는 것도 별개다** — HIGH1(Join 화면 통합 구현)은 design-qa의 관찰 자체는 정확했지만, 사용자가 이미 그 상태를 의도적 스코프로 확정해뒀다는 게 나중에 드러났다. design-qa/design-pl은 "발견"까지만 책임지고, "그게 버그인지 확정된 결정인지"의 최종 판단은 항상 사용자(메인 세션 경유)의 몫으로 남긴다.

**"고쳤는데 절반만 고침" 부작용은 harness-auditor/design-qa 재검증에서 다시 잡힐 수 있고, 같은 승인 범위 안이면 재승인 없이 바로 마무리한다(53차 확립, 63~65차 대량 재확인)**: 65차는 harness-auditor 재검증에서 이전 라운드(64차)가 "정정 지시했다"고 보고한 것 중 실제로 반영 안 된 항목(graphic-assets.md 버전 인용, BgPixels 이력 각주)을 다시 잡아냈다 — 백그라운드 호출 결과를 무비판적으로 "완료"로 받아들이지 않고 재검증까지 마쳐야 한다는 원칙이 강화됐다.
- **64차 원칙(하네스 반영 완료, `token-architecture-guide.md` 6번)**: 색상 틴트/셰이드는 opacity로 파생시키지 않고, 실제 합성 결과 hex를 계산해 독립 Primitive로 등록한다.

**포터블 파일(A범위) leak 정정은 "삭제"가 아니라 "일반 원칙 + 구체 사례는 프로젝트 데이터 문서 참고로 안내"가 정답이다(53차 재확인, 65차 재실행)**: harness-auditor가 `.claude/agents/ui-designer.md`의 "무채색 토큰" 원칙 예시에 박힌 `#ffffff`/`#1a1a1a`(실제 프로젝트 토큰 값과 정확히 일치)를 leak으로 확정 판단해, design-pl이 직접 로컬·전역(`~/.claude/agents/ui-designer.md`) 두 파일 모두에서 구체 hex를 제거하고 "raw 무채색 hex" + "상세 사례는 design-system.md 참고"로 일반화했다(로컬은 프로젝트 경로 그대로, 전역은 "프로젝트별 docs/design/design-system.md"로 한 번 더 일반화). 로컬만 고치고 전역 동기화를 빠뜨리지 않는 습관이 계속 유효했다(과거 전역 파일 자체에 무채색 규칙 문장이 아예 없었던 것도 이번에 발견해 함께 추가).

**QA 감사는 "실제 변경이 있었지만 harness-auditor만 거치고 design-qa를 건너뛴 라운드"를 찾아내 소급 실행할 수 있다(49차 확립)**: 65차에서 harness-auditor A/B범위 소급 검증을 완료했다(64차 미이행분 해소) — 43차 로그로 기록.

**변수 미사용 판정은 "직접 참조 0회"만으로 끝내면 오탐이 난다**: 반드시 두 단계(①노드 직접 바인딩 재귀 스캔 ②전체 변수의 `valuesByMode`에서 VARIABLE_ALIAS 체인 확인)를 다 거쳐야 신뢰할 수 있다.

**❌ 미채택/❌ 폐기 라벨 산출물은 무엇이든(화면·컴포넌트·토큰) 예외 없이 삭제 금지**: 유일한 정당 처리는 컴포넌트 해제(detach→FRAME 전환)뿐이고, 삭제가 필요해 보이면 반드시 메인 세션에 먼저 보고한다.

**하네스 규칙 13번(harness-auditor 호출) — A범위·B범위(`docs/design/**`) 어느 쪽을 건드려도 라운드 종료 전 반드시 호출한다.** (65차 이행 완료 — 43차 로그, 잔여 LOW 2건은 정보성으로 이월)

**"인스턴스 단위 오버라이드"를 결정하기 전 확정 디자인 실측이 선행돼야 한다**: 오버라이드 여부 결정 전 확정 디자인 실측 → 기존 토큰 일치 확인 → 적용 순서를 지킨다.

**단순 값 교체처럼 보여도 아이콘 리바인딩은 반드시 graphic-designer와 design-systems를 순서대로(동시 아님) 함께 부른다(59차·60차 확립)**.

**"새로 조립한 화면에 버튼/토스트/모달이 있으면 interaction-designer 투입은 예외 없이 별도 단계다"(하네스 규칙, 65차 실행 사례로 재확인)**: 이번 세션 신규 화면 8개 전부에 interaction-designer를 투입해 State 상속 확인(대부분 이미 상속) + 신규 전환(Toast 180ms, 모달오픈 250ms, 비밀번호재설정단계전환 300ms) 정의를 마쳤다. 그 과정에서 **raw 목업(mainComponent 없는 텍스트+FRAME) CTA 버튼 1건**(`1060:2014`)을 interaction-designer가 발견 — "화면 조립이 끝났다"와 "그 안의 인터랙티브 요소가 실제 컴포넌트 INSTANCE로 조립됐다"는 별개라는 게 재확인됐다. ui-designer가 즉시 실제 `Button`(`259:609`) INSTANCE로 교체해 해소.
