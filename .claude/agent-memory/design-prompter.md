# design-prompter 메모리

이 파일은 design-prompter가 작업 시작 시 읽고, 작업 종료 시 기록을 남기는 메모리/로그입니다.

## 작업 로그

### 2026-07-12 — brand-designer 브리프: 브랜드 컨셉 시안 3개 (첫 작업)
- **원본 요청**: Figma "레퍼런스" 페이지를 참고해 연락처 관리 웹 서비스의 브랜드 컨셉(톤앤매너) 시안 3개 제작.
- **라우팅**: design-pl이 brand-designer 단독 호출 결정 (design-prompter는 라우팅 변경 없음).
- **시안 단계 판단**: design-pl 판단에 따라 시안 워크플로우 적용 — "브랜드 컨셉 Concepts" 페이지에 시안 3개 + 설명, 사용자 확정 전까지 후속 단계(design-systems 토큰화 등) 금지.
- **구체화 포인트**: (1) 레퍼런스 페이지 get_screenshot 선행 열람 의무화, (2) 3개 시안은 서로 대비되는 방향성이어야 한다는 기준만 제시하고 구체적 컨셉명/컬러는 brand-designer 전문 판단에 위임, (3) 시안 산출물 스코프를 "경량 무드 프레임(컬러 스와치+타이포 샘플)"로 한정해 풀 목업 방지, (4) 실용적 도구 + 교육용 과제 맥락과 3개 간 확실한 차별화 사이 균형점 명시, (5) 페이지/구역 네이밍은 figma-file-organization.md 그대로 따르도록 재확인, (6) 종료 시 agent-memory/brand-designer.md에 시안 3개의 이름/컬러/톤 기록 지시.
- **전달 브리프 원문**: design-pl 응답 메시지 참고 (아래 최종 답변에 포함).

### 2026-07-12 — brand-designer 브리프: 브랜드 컨셉 시안 재작업 (레퍼런스 미확인 버그 수정)
- **원본 요청**: 지난 라운드에서 brand-designer가 최상위 `get_metadata` 목록에 "레퍼런스" 페이지가 안 보인다는 이유로 "없다"고 단정 → 실제로는 nodeId `15:2`에 존재. 도구 한계(최상위 목록이 전체 페이지를 다 반환하지 않을 수 있음)로 인한 오판이었음이 확인됨.
- **라우팅**: design-pl이 brand-designer 단독 재호출 결정 (design-prompter는 라우팅 변경 없음).
- **재작업 성격 규정(핵심 구체화 포인트)**: "사용자가 시안을 거부해서" 재작업하는 게 아니라 "레퍼런스를 못 보고 만든 버그"라서 재작업하는 것임을 브리프에 명시. 이 때문에 `figma-file-organization.md`의 일반 시안 폐기 규칙(❌ 미채택 접두어로 보존)을 이번엔 적용하지 않고, 기존 프레임 34:3/34:4/34:5를 그대로 덮어써도 된다고 예외를 명시 — brand-designer가 규칙을 혼동해 불필요하게 미채택 표시를 남기지 않도록 함.
- **구체화 포인트**: (1) nodeId `15:2`로 직접 get_metadata → 내부 이미지 노드 get_screenshot까지 순차 실행해 실제 열람 의무화(목록 재조회 금지, 이미 nodeId를 알고 있으므로 직접 조회), (2) "레퍼런스를 실제로 봤다"는 근거(확인한 이미지/톤 요약)를 최종 보고에 포함하도록 명시, (3) 확인한 무드보드를 "연락처 관리 앱" 성격에 맞게 재해석해 서로 뚜렷이 구분되는 톤 3개 산출(산출 스코프는 이전과 동일: 컨셉명+설명+주조/보조/강조 컬러+경량 무드 프레임, 풀 목업 아님), (4) 여전히 미확정 상태 유지 — design-pl 확정 전 design-systems 등 후속 단계 진행 금지 재확인, (5) agent-memory/brand-designer.md 갱신 지시(레퍼런스 실제 확인 여부, 새 시안 내용, 미확정 상태), (6) design-pl 보고 순서를 "레퍼런스 실제 열람 여부/내용 → 새 시안 3개 요약" 순으로 고정해 이번 재작업의 목적(레퍼런스 확인 여부 검증)이 보고에서 누락되지 않게 함.
- **전달 브리프 원문**: 아래 최종 답변에 포함.

### 2026-07-12 — ui-designer 브리프: 브랜드 컨셉 3개를 메인 화면(연락처 목록)에 적용한 시안 3장
- **원본 요청**: 사용자가 "무드 프레임만으로는 컨셉을 선택하기 어렵다, 실제 메인 화면에 적용된 모습을 봐야 한다"고 요청. design-pl이 ui-designer 단독 호출 결정(design-systems는 이번 라운드 미투입).
- **라우팅**: design-pl 결정 그대로 따름(변경 없음). design-systems를 부르지 않은 이유(아직 미확정 컨셉이라 정식 토큰화 불가)를 브리프에 명시해 ui-designer가 임의로 토큰 작업을 확장하지 않도록 가드.
- **시안 단계 판단**: PL 판단대로 "이 3장도 확정 아님" 유지 — 컨셉 확정 전 단계이므로 3장 산출 후 사용자 확정 전까지 design-systems 토큰화/고퀄리티 디벨롭으로 진행 금지를 명시.
- **구체화 포인트**:
  1. `.claude/agent-memory/brand-designer.md`의 정확한 hex/폰트 값을 그대로 사용하도록 표로 재확인해서 전달(임의 신규 색상 생성 방지).
  2. 배치 위치를 nodeId까지 못박음 — 새 페이지 생성 금지, 기존 34:2 페이지 안에 34:3/34:4/34:5 브랜드 프레임을 건드리지 않고 그 옆에 각 컨셉 대응 메인 화면 프레임을 추가.
  3. 화면 스코프를 "연락처 목록 메인 화면(헤더/검색/리스트/카테고리)"으로 한정해 로그인·CRUD 모달 등 다른 화면까지 확장하지 않도록 범위 제한.
  4. 각 프레임에 1~2문장 설명 부착 의무화(figma-file-organization.md 시안 워크플로우 규칙 그대로 적용).
  5. 완료 기준에 "브랜드 프레임 34:3/34:4/34:5 원본 유지 여부"를 명시적으로 넣어, 실수로 기존 컨셉 프레임을 수정/덮어쓰는 사고를 방지.
  6. design-pl 보고 형식을 "프레임별 nodeId + 배치 요소 + 적용한 컨셉 색/폰트" 순으로 고정.
- **전달 브리프 원문**: 아래 최종 답변에 포함.

### 2026-07-12 — brand-designer 브리프: Concept B "Warm Ledger" 확정 후 정식화 (Brand Guide 완성 + 미채택 라벨링)
- **원본 요청**: 사용자가 브랜드 컨셉 시안 3개(Deskline/Warm Ledger/Gridline) 중 Concept B "Warm Ledger"를 최종 확정. design-pl이 brand-designer 단독 호출 결정(시안 단계 종료, 정식화 단계로 진입).
- **라우팅**: design-pl 결정 그대로 따름(변경 없음). 이번 호출은 시안 제작이 아니라 "확정 → 정식화 + 미채택 정리"이므로 시안 3개 재생성 금지를 브리프에 명시.
- **시안 단계 판단**: PL 판단대로 시안 워크플로우 종료 단계. figma-file-organization.md 2번 4-(a)의 표준 파이프라인 중 (a) brand-designer 정식화 단계만 이번 라운드 범위이고, (b) design-systems 토큰화·(c) ui-designer SCREENS 정식 개발은 다음 라운드(design-pl이 후속 호출)임을 명시해 brand-designer가 임의로 앞서 나가지 않도록 가드.
- **구체화 포인트**:
  1. Concept B의 정확한 확정값(주조 #17A398 / 보조 #FF5A76 / 강조 #FFCB47, Baloo 2 + Noto Sans KR Black/Regular/Medium)을 `.claude/agent-memory/brand-designer.md` 기존 표에서 그대로 가져오도록 못박아, 정식화 과정에서 값이 미세하게 바뀌는 사고 방지.
  2. **워드마크 네이밍 모호함 선제 차단(design-prompter가 메모리에서 발견해 추가한 지점)**: 기존 34:4 프레임의 "yourbook." 등은 brand-designer 메모리에 명시적으로 "최종 네이밍이 아닌 톤 데모용 placeholder"라고 기록되어 있음. Brand Guide 정식화 시 이 placeholder를 최종 서비스명으로 임의 확정하지 말고, 로고/워드마크 사용 예시는 서체·톤 표현 방식(굵은 아웃라인, 축소/확대 시 무너짐 여부)에 집중하며 실제 서비스명이 필요하면 "제품명 미정 — placeholder 사용, 확정 시 design-pl 경유 별도 논의 필요"라고 Brand Guide에 주석으로 남기도록 지시. 이렇게 하지 않으면 brand-designer가 재량으로 서비스명을 확정해버려 사용자 의도와 다른 산출물이 나올 위험이 있었음.
  3. 미채택 라벨링 대상을 프레임 ID로 못박음: 34:3(Concept A 브랜드), 44:2(Concept A 적용 화면), 34:5(Concept C 브랜드), 46:2(Concept C 적용 화면) — 이름 앞에 `❌ 미채택 —`만 추가, 내부 디자인 절대 수정 금지. 44:2/46:2는 ui-designer가 만든 프레임이지만 라벨링은 정식화 담당(brand-designer)이 세트 전체에 처리한다는 2번 5항 규칙을 명시해 "내가 안 만든 프레임이니 손대면 안 된다"는 혼동 방지.
  4. 확정된 34:4/45:2(Concept B 브랜드/적용 화면)는 라벨 변경 없이 그대로 유지 — Concepts 페이지는 의사결정 히스토리 보존용이고, 정식 자산은 별도로 "Brand Guide" 페이지에 이관된다는 점을 명확히 구분.
  5. 완료 기준에 "34:4/45:2는 변경 없음"과 "34:3/44:2/34:5/46:2는 라벨만 붙고 내용 그대로"를 둘 다 명시적으로 넣어 검증 가능하게 함.
  6. design-pl 보고 형식을 "Brand Guide 페이지 nodeId → 확정한 내용 → 라벨링 처리 결과" 순으로 고정.
- **전달 브리프 원문**: 아래 최종 답변에 포함.

### 2026-07-12 — ui-designer 브리프: Warm Ledger 확정 후 "연락처 목록" 화면 레이아웃 시안 A/B/C 재작업 (기획문서 요건 반영, placeholder 버그 수정)
- **원본 요청**: 사용자가 "Warm Ledger 컨셉으로 연락처 목록 화면 디자인 시안 A/B/C를 다시 뽑아줘 — 이번엔 기획문서 요건을 반영해서"라고 요청. design-pl이 ui-designer 단독 호출 결정(design-systems 이번 라운드 미투입).
- **라우팅**: design-pl 결정 그대로 따름(변경 없음).
- **재작업 배경(핵심 구체화 포인트)**: 3차 라운드에서 만든 `45:2`(Concept B 적용 화면)가 기획 문서를 전혀 반영하지 않은 "검색창+목업 연락처 7개" 제네릭 placeholder였다는 걸 사용자가 지적. 방금 `ui-designer.md`에 "시안 단계에서도 요소는 placeholder가 아니라 기획 문서의 실제 요소를 반영해야 한다"는 규칙이 막 보강된 직후의 첫 실사례라는 점을 브리프에 명시해, ui-designer가 이 규칙을 가볍게 여기지 않고 최우선으로 지키게 함.
- **라운드 성격 판단**: design-pl 판단은 Type (C) "화면 레이아웃만" 라운드에 해당하나, 정상적인 Type C 전제(토큰/컴포넌트 기확보)가 아직 안 갖춰진 상태 — brand-designer 확정은 됐지만 design-systems 토큰화가 아직 없음. 따라서 `ui-designer.md`의 "브랜드 확정 전 단계" 하드코딩 예외를 이번에도 그대로 적용하도록 명시(design-systems 투입 없이 Warm Ledger 원본 hex/폰트를 ui-designer가 직접 사용). PL이 이미 판단한 라우팅이므로 design-prompter가 별도로 바꾸지 않음.
- **구체화 포인트**:
  1. Warm Ledger 확정값(주조 #17A398/보조 #FF5A76/강조 #FFCB47 · 60:30:10, Baloo 2 + Noto Sans KR Black/Regular/Medium, "다정하지만 칠칠맞지 않은 친구" 퍼스낼리티·24px 이상 굵은 잉크 아웃라인으로 "내 목록 vs 남의 목록" 데이터 경계 표현)을 brand-designer.md 표 그대로 못박아 임의 조정 금지.
  2. **기획 문서 실제 열람을 절차로 강제**: `docs/planning`의 화면정의서(`02_...화면정의서_v1.0.pdf`)·기능정의서(`03_...기능정의서_v1.0.pdf`), 필요시 구현요구사항(`01_...`)/PRD(`04_...`)를 Read 도구로 직접 읽고 "연락처 목록" 화면 정의를 찾아 반영하도록 지시(문서 열람 없이 만들면 이전과 같은 placeholder 버그 재발). 최소 반영 요소를 6개 항목(검색/필터 조건, 목록 표시 필드, 카테고리/그룹 표현, 액션 진입점, 상태 표현(빈 목록/사용자별 데이터 격리), placeholder 절대 금지+프레임 설명에 문서 근거 명시)으로 구체적으로 못박아 "기획문서 반영"이라는 추상적 지시가 다시 뭉뚱그려지지 않게 함. 문서에 없는 내용은 임의로 지어내지 말라고 명시(과잉구현 방지).
  3. **레이아웃 3안이 실질적으로 달라야 함을 구체 축으로 명시**: 네비게이션 위치(상단바/좌측 사이드바), 리스트 표현 방식(카드형/테이블형/그리드형), 정보 밀도 — ui-designer.md의 레이아웃 구조 다양성 원칙을 그대로 재확인해 색만 다른 3장 재발 방지.
  4. **배치 위치와 재작업 성격을 명시적으로 구분**: 대상은 `34:2` 페이지의 `45:2`(Concept B 적용 화면) 자리이며, 이는 브랜드 재선택이 아니라 부실 결과물의 버그성 재작업이므로 `❌ 미채택 —` 라벨 없이 덮어쓰라고 명시(3차 라운드의 "레퍼런스 미확인 재작업" 전례와 동일 처리 원칙 인용). `44:2`/`46:2`(미채택 컨셉 A/C 적용 화면)는 이번 작업과 무관하니 절대 건드리지 않는다고 별도로 못박아, "A/B/C 시안"이라는 표현 때문에 ui-designer가 기존 미채택 프레임까지 건드리는 혼동을 사전 차단.
  5. 각 프레임에 (1) 반영한 기획 문서 요소, (2) 3개 레이아웃 구조 차이를 설명 텍스트로 남기도록 의무화(figma-file-organization.md 시안 워크플로우 규칙과 동일 적용).
  6. 시안 단계이므로 사이즈 제약(1440px 등 정식 SCREENS 기준)은 이번 대상이 아니라고 명시하되, 기획 요소 실제 반영은 시안 단계에도 예외 없이 적용됨을 재차 구분해 "가벼운 목업이니 대충 해도 된다"는 오독 방지.
